# 15. 데이터 타입

## 15.1 문자열(CHAR와 VARCHAR)

### 15.1.1 저장 공간

- CHAR와 VARCHAR의 가장 큰 차이는 고정 길이냐 가변 길이냐이다.
- CHAR는 실제 저장값이 짧아도 정의된 길이만큼 공간을 사용한다.
    
    <img width="466" height="143" alt="image" src="https://github.com/user-attachments/assets/84378fc6-7ad3-4ecc-a1e3-98e2257d855a" />

- VARCHAR는 실제 저장된 값의 길이에 따라 필요한 공간만 사용한다. 대신 저장된 길이 정보를 함께 보관해야 하므로 1~2바이트의 추가 공간이 필요하다.
    
    <img width="348" height="146" alt="image" src="https://github.com/user-attachments/assets/f416574a-d8b7-445a-9376-09b3301e156b" />

- 칼럼 하나만 보면 VARCHAR가 더 효율적으로 보일 수 있지만, 값 길이 변화가 잦은 경우에는 다른 비용이 생긴다.
- 레코드 중간의 VARCHAR 칼럼 길이가 커지면 뒤쪽 칼럼의 저장 위치가 밀릴 수 있다. 이 경우 레코드 재배치나 페이지 내 이동이 발생할 수 있다.
- CHAR는 이런 위치 이동이 적지만, 짧은 값이 많으면 남는 공간이 많아질 수 있다.
- 한글과 같은 멀티바이트 문자는 문자 수와 바이트 수가 다르므로, 길이 정의와 실제 저장 공간을 구분해서 봐야 한다.
- ASCII 계열은 보통 문자당 1바이트, 한글 완성형 계열은 2바이트, UTF-8 계열은 최대 3바이트, utf8mb4는 최대 4바이트까지 사용할 수 있다.
- 칼럼 길이를 정할 때의 핵심 기준은 다음 2가지다.
    - 저장되는 문자열 길이가 대체로 고정적인가
    - 칼럼 값이 자주 변경되는가
- 값 길이가 거의 일정하면 CHAR가 유리하고, 길이 편차가 크면 VARCHAR가 일반적으로 적합하다.

### 15.1.2 저장 공간과 스키마 변경(Online DDL)

- VARCHAR 칼럼의 길이를 늘리는 작업은 단순 길이 확장처럼 보여도 실제 저장 공간 구조와 연결된다.
- 특히 멀티바이트 문자 집합에서는 정의 길이 변화가 레코드 최대 길이 구조를 바꾸기 때문에, 어떤 변경은 빠르게 끝나고 어떤 변경은 테이블 재구성이 필요하다.
- 예시에서 `VARCHAR(60)`을 `VARCHAR(63)`으로 바꾸는 것은 INPLACE와 LOCK=NONE으로 처리되지만, `VARCHAR(64)`로 늘리는 순간 COPY 방식이 필요해졌다.
- 이유는 길이 정보를 저장하는 내부 구조가 바뀌기 때문이다. 최대 저장 바이트 수가 특정 경계를 넘으면 길이 저장 바이트 수가 달라진다.
- 이 경계를 넘는 변경은 단순 메타데이터 수정이 아니라 실제 데이터 재배치가 필요할 수 있다.
- 따라서 처음 설계할 때 VARCHAR 길이를 너무 타이트하게 잡지 말고, 예상 증가 폭을 고려해 약간 여유 있게 정의하는 편이 좋다.
- 서비스 중 스키마 변경 비용을 줄이기 위해서도 초기에 길이 설계가 중요하다.

### 15.1.3 문자 집합(캐릭터 셋)

- MySQL은 칼럼, 테이블, 데이터베이스, 서버 단위로 문자 집합을 설정할 수 있다.
- 한국어 환경에서는 주로 `euckr` 또는 `utf8mb4`를 사용한다.
- 여러 언어를 함께 저장해야 하거나 웹 서비스 중심이라면 `utf8mb4`가 사실상 기본 선택지에 가깝다.
- `utf8`은 MySQL에서 과거의 3바이트 UTF-8 구현이며, 모든 유니코드 문자를 담을 수 없다. 이모지나 일부 확장 문자는 저장하지 못한다.
- `utf8mb4`는 최대 4바이트까지 저장할 수 있어 전체 유니코드 지원을 목표로 한다.
- `utf8mb4`가 더 많은 문자를 담을 수 있지만 저장 공간과 인덱스 길이에는 더 민감하다.
- 문자 집합 관련 시스템 변수는 각각 역할이 다르다.
    - `character_set_system`: 시스템 내부 기본 문자 집합
    - `character_set_server`: 서버 기본 문자 집합
    - `character_set_database`: DB 기본 문자 집합
    - `character_set_filesystem`: 파일명 해석용 문자 집합
    - `character_set_client`: 클라이언트가 보내는 SQL 문장의 문자 집합
    - `character_set_connection`: 클라이언트 입력값을 내부 처리용으로 변환할 때의 문자 집합
    - `character_set_results`: 결과를 클라이언트에 돌려줄 때의 문자 집합
        
        <img width="679" height="497" alt="image" src="https://github.com/user-attachments/assets/bcb29f28-e65f-43c6-833b-54c79bb825bb" />

- 15.1.3.1 클라이언트로부터 쿼리를 요청했을 때의 문자 집합 변환
    - 서버는 클라이언트에서 전달된 SQL 문장을 `character_set_client` 기준으로 해석하고, 필요하면 `character_set_connection` 기준으로 변환해 처리한다.
    - 리터럴 앞에 `_latin1`, `_utf8mb4` 같은 introducer를 붙이면 해당 문자열 상수의 문자 집합을 명시할 수 있다.
    - 이 값은 칼럼 문자 집합과 다를 수 있으며, 비교 시 변환 규칙의 영향을 받는다.
- 15.1.3.2 처리 결과를 클라이언트로 전송할 때의 문자 집합 변환
    - 실행 결과는 `character_set_results` 기준으로 변환되어 반환된다.
    - `SET NAMES utf8mb4`는 `character_set_client`, `character_set_connection`, `character_set_results`를 한 번에 맞추는 효과가 있다.
    - `CHARSET utf8mb4`도 비슷한 목적이지만 적용 범위와 동작 타이밍에 차이가 있다.
    - 클라이언트와 서버의 문자 집합이 어긋나면 저장은 정상인데 화면 출력만 깨지는 문제도 생길 수 있다.

### 15.1.4 콜레이션(Collation)

- 콜레이션은 문자열의 비교와 정렬 규칙이다.
- 같은 문자 집합이라도 콜레이션이 다르면 대소문자 구분 여부, 악센트 구분 여부, 정렬 순서가 달라진다.
- 콜레이션 이름은 보통 문자 집합명과 성격을 함께 담는다.
    - `_ci`: 대소문자 비구분
    - `_cs`: 대소문자 구분
    - `_bin`: 바이너리 비교
- utf8mb4 계열에서는 UCA 버전에 따른 차이도 중요하다.
- `utf8mb4_general_ci`는 오래된 단순 규칙 기반이고, `utf8mb4_0900_ai_ci`는 UCA 9.0 기준을 반영한다.
- `ai`는 악센트 비구분, `as`는 악센트 구분을 의미한다.
- 대소문자만이 아니라 악센트와 문자 정규화까지 비교 결과에 영향을 준다.
- MySQL 8.0에서는 기본 콜레이션이 `utf8mb4_0900_ai_ci` 계열로 바뀌었다.
- MySQL 5.7에서 8.0으로 업그레이드하면 기존 객체와 신규 객체의 기본 콜레이션이 달라질 수 있으므로 조인, 비교, 정렬 시 오류나 혼합 문제가 발생할 수 있다.
- 연결 파라미터에 `connectionCollation` 같은 설정을 추가해 애플리케이션 연결 시점의 콜레이션을 맞추는 것이 중요하다.
- 15.1.4.1 콜레이션 이해
    - 같은 칼럼 타입이라도 콜레이션에 따라 `a`, `A`, `á` 등의 비교 결과와 정렬 결과가 달라진다.
    - 테이블, 칼럼, 리터럴이 서로 다른 콜레이션을 가지면 비교 시 변환 규칙이 추가된다.
    - 비교와 정렬이 많은 칼럼은 문자 집합만이 아니라 콜레이션까지 설계 대상이다.
    - `information_schema.columns`에서 칼럼별 문자 집합과 콜레이션을 확인할 수 있다.
- 15.1.4.2 utf8mb4 문자 집합의 콜레이션
    - `utf8mb4_general_ci`는 단순하고 빠르지만 정교함은 떨어진다.
    - `utf8mb4_0900_ai_ci`는 더 정교한 비교 규칙을 제공하지만 상대적으로 무겁다.
    - 성능만 보고 무조건 `general_ci`를 선택할 것이 아니라, 검색 정확도와 정렬 규칙 요구사항을 같이 봐야 한다.
    - 서비스에서 국제화 요구가 크면 최신 UCA 기반 콜레이션이 더 적합할 수 있다.

### 15.1.5 비교 방식

- MySQL의 문자열 비교는 문자 집합과 콜레이션의 영향을 직접 받는다.
- CHAR 타입에서는 끝 공백 처리도 비교에 영향을 준다.
- `'ABC' = 'ABC '` 비교는 공백 무시 규칙 때문에 참이 될 수 있다.
- 하지만 `LIKE` 비교는 일반 비교와 다르게 동작할 수 있으며, 공백을 더 민감하게 다룬다.
- 동일 문자열 비교, 정렬, LIKE 검색의 결과가 모두 같을 것이라고 가정하면 안 된다.
- 콜레이션이 섞이면 `Illegal mix of collations` 오류가 날 수 있다.
- 특히 업그레이드 과정에서 기존 테이블은 옛 콜레이션, 신규 연결은 새 콜레이션을 쓰는 상황을 주의해야 한다.

### 15.1.6 문자열 이스케이프 처리

- MySQL은 문자열 안의 특수 문자를 `\`로 이스케이프할 수 있다.
- 대표적으로 `\0`, `\'`, `\"`, `\b`, `\n`, `\r`, `\t`, `\\`, `\%`, `\_` 등이 있다.
- `LIKE` 검색에서는 `%`와 `_`가 패턴 문자이므로 실제 문자로 검색하려면 이스케이프가 필요하다.
- 홑따옴표와 쌍따옴표는 SQL 모드와 문자열 표기 방식에 따라 해석이 달라질 수 있다.
- 백슬래시를 포함한 문자열은 클라이언트, SQL 모드, 애플리케이션 언어의 문자열 리터럴 규칙이 함께 얽히므로 주의해야 한다.
- 문자열 검색 오류나 예상과 다른 매칭이 생기면 실제 저장값과 이스케이프 규칙을 함께 확인해야 한다.

## 15.2 숫자

- 숫자 타입은 값의 정확도 기준으로 참값과 근삿값 타입으로 나뉜다.
- 참값은 소수점 이하 포함 여부와 관계없이 저장한 값을 그대로 유지하는 타입이다. INTEGER 계열과 DECIMAL이 해당한다.
- 근삿값은 저장 시점과 조회 시점 값이 정확히 일치하지 않을 수 있는 타입이다. FLOAT와 DOUBLE이 해당한다.
- 저장 포맷 기준으로는 이진 표기법과 십진 표기법으로 나뉜다.
    - 이진 표기법은 정수나 실수를 이진 기반으로 저장하는 방식이다. MySQL의 INTEGER와 BIGINT 등 대부분 숫자 타입이 여기에 해당한다.
    - 십진 표기법은 숫자의 각 자릿값을 4비트 단위로 표현하는 방식이며 DECIMAL이 사용한다.
- 복제나 정확한 비교가 중요하면 FLOAT나 DOUBLE보다 INTEGER 계열이나 DECIMAL을 우선 고려하는 편이 안전하다.
- 매우 큰 숫자나 고정 소수점이 필요하지 않다면 일반적으로 INTEGER나 BIGINT를 더 자주 사용한다.

### 15.2.1 정수

- 정수 저장용 타입은 TINYINT, SMALLINT, MEDIUMINT, INT, BIGINT 다섯 가지가 있다. 차이는 주로 저장 공간과 저장 가능한 값 범위다.
    
    <img width="586" height="192" alt="image" src="https://github.com/user-attachments/assets/0a7f23f4-fdc8-4bd1-978f-7e1827427ffc" />

- 정수 칼럼은 UNSIGNED 옵션을 사용할 수 있다.
    - 기본은 SIGNED이며 음수와 양수를 함께 저장한다.
    - UNSIGNED를 사용하면 음수는 저장하지 못하지만 더 큰 양수 범위를 저장할 수 있다.
- 조인 시 SIGNED와 UNSIGNED 칼럼이 섞여도 인덱스 사용 여부 자체가 깨지는 것은 아니지만, 저장 범위가 다르므로 외래 키나 조인 조건에서는 옵션을 맞추는 편이 좋다.

### 15.2.2 부동 소수점

- 부동 소수점은 소수점의 위치가 고정되지 않기 때문에 정확한 유효 소수점 값을 식별하거나 정확 비교를 수행하기 어렵다.
- FLOAT는 기본적으로 4바이트를 사용해 유효 자릿수 8개 정도를 유지하고, DOUBLE은 8바이트를 사용해 유효 자릿수 16개 정도를 유지한다.
- STATEMENT 기반 복제 환경에서는 부동 소수점 값이 소스와 레플리카에서 다르게 저장될 수 있어 특히 주의해야 한다.
- 좌표처럼 소수 자릿수만 유지하면 되는 값은 소수점을 없애 정수로 변환해 저장하는 방법도 있다.
    - 예시처럼 위도, 경도에 10000을 곱해 INT에 저장하고, 조회 시 다시 10000으로 나누는 방식이다.

### 15.2.3 DECIMAL

- 금액이나 이자처럼 소수점 이하까지 정확히 유지해야 하는 값은 FLOAT나 DOUBLE이 아니라 DECIMAL을 사용해야 한다.
- DECIMAL은 숫자 두 자리를 저장하는 데 1바이트가 필요하므로 저장 공간은 대체로 정수 타입보다 더 많이 든다.
- DECIMAL과 BIGINT의 곱셈 성능을 비교하면 BIGINT가 조금 더 빠르다.
- 정수만 저장할 것인데 DECIMAL을 쓰는 것은 저장 공간과 성능 면에서 유리하지 않다.

### 15.2.4 정수 타입 칼럼을 생성할 때의 주의사항

- DECIMAL(M,D)은 전체 자릿수 M, 소수부 자릿수 D를 의미한다.
- BIGINT(10)처럼 작성해도 실제로는 BIGINT 자체의 저장 범위가 사용된다.

### 15.2.5 자동 증가(AUTO_INCREMENT) 옵션 사용

- AUTO_INCREMENT는 인조 키를 만들 때 사용하는 옵션이다.
- `auto_increment_increment`와 `auto_increment_offset` 시스템 변수로 증가 패턴을 바꿀 수 있다.
    - 예를 들어 increment=10, offset=5면 5, 15, 25 순으로 증가한다.
- AUTO_INCREMENT 칼럼은 반드시 프라이머리 키나 유니크 키의 일부여야 한다.
- InnoDB에서는 AUTO_INCREMENT 칼럼으로 시작하는 인덱스가 필요하다.
    - 프라이머리 키 뒤쪽에 AUTO_INCREMENT 칼럼을 배치하면 오류가 발생할 수 있다.
- AUTO_INCREMENT 값은 테이블당 하나만 사용할 수 있다.

## 15.3 날짜와 시간

- MySQL은 날짜와 시간을 분리 저장하거나 하나의 칼럼에 합쳐 저장할 수 있는 여러 타입을 제공한다.
- YEAR, DATE, TIME, DATETIME, TIMESTAMP가 대표적이다.

### 15.3.1 자동 업데이트

- MySQL 5.6 이전에는 TIMESTAMP 칼럼만 자동 초기화와 자동 업데이트 규칙이 복잡하게 적용됐다.
- MySQL 5.6부터는 TIMESTAMP와 DATETIME 모두 `DEFAULT CURRENT_TIMESTAMP`, `ON UPDATE CURRENT_TIMESTAMP`를 사용할 수 있다.
- INSERT 시점에는 기본값이 들어가고, UPDATE 시점에는 지정된 칼럼이 자동으로 현재 시각으로 바뀐다.
- 이제 DATETIME과 TIMESTAMP의 차이는 자동 업데이트 규칙보다 타임존 처리 방식에 더 크게 있다.

### 15.3.2 DATETIME과 TIMESTAMP의 차이

- DATETIME은 칼럼에 타임존 정보가 저장되지 않는다.
- DATETIME은 클라이언트가 입력한 값을 그대로 저장하고 조회 시에도 변환 없이 반환한다.
- TIMESTAMP는 항상 UTC로 저장되고, 조회 시에는 현재 세션의 타임존에 맞게 변환되어 보인다.
- 같은 NOW() 값을 넣더라도 세션 타임존을 바꾸면 DATETIME은 값이 그대로이고 TIMESTAMP는 다르게 보인다.
- 타임존 정보가 MySQL 서버에 적재되지 않은 경우 named timezone 설정은 실패할 수 있다. 이때는 `+09:00` 같은 offset 형식 타임존을 사용할 수 있다.
- JDBC 애플리케이션에서는 JVM 타임존, JDBC 연결 문자열의 `serverTimezone`, MySQL 세션 타임존이 함께 영향을 준다.
    - JDBC 드라이버는 DATETIME도 애플리케이션 레벨에서 타임존 변환된 것처럼 보이게 만들 수 있으므로 실제 동작을 반드시 테스트해야 한다.
- ORM을 사용하는 환경에서는 프레임워크가 날짜 타입을 자동 매핑하므로, JDBC API 레벨에서 어떤 값으로 패치되는지 미리 검증하는 것이 중요하다.
- MySQL 서버의 기본 타임존은 `system_time_zone`, 세션 타임존은 `time_zone` 시스템 변수로 확인한다.
    - 세션의 NOW()나 TIMESTAMP 처리에는 `time_zone`이 직접 영향을 준다.
    - `time_zone=SYSTEM`이면 `system_time_zone` 값을 따른다.
- 이미 저장된 DATETIME 값을 다른 타임존 기준으로 바꾸려면 서버 타임존 변경만으로는 해결되지 않고 `CONVERT_TZ()` 같은 변환 함수가 필요하다.
- 반대로 TIMESTAMP는 서버에 UTC로 저장되므로 서버 타임존 변경 때문에 저장값 자체를 별도 변환할 필요는 없다.

## 15.4 ENUM과 SET

### 15.4.1 ENUM

- ENUM은 미리 정의한 목록 중 하나의 값만 저장하는 타입이다.
- 저장 시에는 사용자가 입력한 문자열이 아니라 내부적으로 매핑된 정수값이 저장된다.
- 아이템 개수가 255개 미만이면 1바이트, 그 이상이면 2바이트 저장 공간을 사용한다.
- 매핑된 정수값은 정의 순서대로 1부터 시작하며 빈 문자열은 0으로 매핑된다.
- ENUM 칼럼은 문자처럼 비교하고 조회할 수 있지만, 정렬 시에는 문자값이 아니라 내부 정수 코드 순서로 정렬된다.
- 문자 기준으로 정렬하려면 `CAST(... AS CHAR)`가 필요하지만, 이 경우 인덱스를 이용한 정렬은 기대하기 어렵다.
- MySQL 5.6부터는 새 아이템을 ENUM 정의의 마지막에 추가하는 경우 메타데이터 변경만으로 끝날 수 있다.
- 중간에 값을 끼워 넣거나 순서를 바꾸면 COPY 알고리즘이 필요해지고 읽기 잠금이 걸릴 수 있다.
- ENUM의 장점은 저장 공간 절감이다.
    - 문자 코드를 그대로 저장하는 것보다 작은 공간을 쓰므로 디스크 사용량, 백업 파일 크기, 인덱스 크기를 줄이는 효과가 있다.

### 15.4.2 SET

- SET은 미리 정의한 아이템 중 여러 개를 동시에 선택해 저장할 수 있는 타입이다.
- 내부적으로는 비트 연산 기반 정수값으로 저장된다.
    - 아이템이 여러 개 선택되면 각 아이템 비트를 OR 해서 저장한다.
- 아이템 개수에 따라 1바이트부터 최대 8바이트까지 저장 공간을 사용한다.
- 여러 값을 저장할 때는 콤마로 구분한 문자열 형태로 입력하고, 조회 시에도 콤마로 이어진 문자열처럼 보인다.
- 특정 값 포함 여부는 `FIND_IN_SET()` 또는 LIKE 검색으로 확인할 수 있다.
- 다만 `FIND_IN_SET()`이나 LIKE 조건은 인덱스가 있어도 효율적으로 활용하기 어렵다.
- 이런 검색이 자주 사용되면 SET 대신 정규화한 별도 자식 테이블 구조가 더 적합하다.
- SET도 ENUM과 비슷하게 새 아이템을 추가할 때 구조 변경이 필요하다.
    - 마지막에 새 아이템을 붙이는 경우 메타데이터 변경만으로 끝날 수 있지만,
    - 저장 비트 수가 바뀌는 구간, 예를 들어 8개에서 9개로 늘어나는 경우에는 리빌드가 필요하다.

# 15. 데이터 타입

## 15.1 문자열(CHAR와 VARCHAR)

### 15.1.1 저장 공간

- CHAR와 VARCHAR의 가장 큰 차이는 고정 길이냐 가변 길이냐이다.
- CHAR는 실제 저장값이 짧아도 정의된 길이만큼 공간을 사용한다.
    
    <img width="466" height="143" alt="image" src="https://github.com/user-attachments/assets/84378fc6-7ad3-4ecc-a1e3-98e2257d855a" />

- VARCHAR는 실제 저장된 값의 길이에 따라 필요한 공간만 사용한다. 대신 저장된 길이 정보를 함께 보관해야 하므로 1~2바이트의 추가 공간이 필요하다.
    
    <img width="348" height="146" alt="image" src="https://github.com/user-attachments/assets/f416574a-d8b7-445a-9376-09b3301e156b" />

- 칼럼 하나만 보면 VARCHAR가 더 효율적으로 보일 수 있지만, 값 길이 변화가 잦은 경우에는 다른 비용이 생긴다.
- 레코드 중간의 VARCHAR 칼럼 길이가 커지면 뒤쪽 칼럼의 저장 위치가 밀릴 수 있다. 이 경우 레코드 재배치나 페이지 내 이동이 발생할 수 있다.
- CHAR는 이런 위치 이동이 적지만, 짧은 값이 많으면 남는 공간이 많아질 수 있다.
- 한글과 같은 멀티바이트 문자는 문자 수와 바이트 수가 다르므로, 길이 정의와 실제 저장 공간을 구분해서 봐야 한다.
- ASCII 계열은 보통 문자당 1바이트, 한글 완성형 계열은 2바이트, UTF-8 계열은 최대 3바이트, utf8mb4는 최대 4바이트까지 사용할 수 있다.
- 칼럼 길이를 정할 때의 핵심 기준은 다음 2가지다.
    - 저장되는 문자열 길이가 대체로 고정적인가
    - 칼럼 값이 자주 변경되는가
- 값 길이가 거의 일정하면 CHAR가 유리하고, 길이 편차가 크면 VARCHAR가 일반적으로 적합하다.

### 15.1.2 저장 공간과 스키마 변경(Online DDL)

- VARCHAR 칼럼의 길이를 늘리는 작업은 단순 길이 확장처럼 보여도 실제 저장 공간 구조와 연결된다.
- 특히 멀티바이트 문자 집합에서는 정의 길이 변화가 레코드 최대 길이 구조를 바꾸기 때문에, 어떤 변경은 빠르게 끝나고 어떤 변경은 테이블 재구성이 필요하다.
- 예시에서 `VARCHAR(60)`을 `VARCHAR(63)`으로 바꾸는 것은 INPLACE와 LOCK=NONE으로 처리되지만, `VARCHAR(64)`로 늘리는 순간 COPY 방식이 필요해졌다.
- 이유는 길이 정보를 저장하는 내부 구조가 바뀌기 때문이다. 최대 저장 바이트 수가 특정 경계를 넘으면 길이 저장 바이트 수가 달라진다.
- 이 경계를 넘는 변경은 단순 메타데이터 수정이 아니라 실제 데이터 재배치가 필요할 수 있다.
- 따라서 처음 설계할 때 VARCHAR 길이를 너무 타이트하게 잡지 말고, 예상 증가 폭을 고려해 약간 여유 있게 정의하는 편이 좋다.
- 서비스 중 스키마 변경 비용을 줄이기 위해서도 초기에 길이 설계가 중요하다.

### 15.1.3 문자 집합(캐릭터 셋)

- MySQL은 칼럼, 테이블, 데이터베이스, 서버 단위로 문자 집합을 설정할 수 있다.
- 한국어 환경에서는 주로 `euckr` 또는 `utf8mb4`를 사용한다.
- 여러 언어를 함께 저장해야 하거나 웹 서비스 중심이라면 `utf8mb4`가 사실상 기본 선택지에 가깝다.
- `utf8`은 MySQL에서 과거의 3바이트 UTF-8 구현이며, 모든 유니코드 문자를 담을 수 없다. 이모지나 일부 확장 문자는 저장하지 못한다.
- `utf8mb4`는 최대 4바이트까지 저장할 수 있어 전체 유니코드 지원을 목표로 한다.
- `utf8mb4`가 더 많은 문자를 담을 수 있지만 저장 공간과 인덱스 길이에는 더 민감하다.
- 문자 집합 관련 시스템 변수는 각각 역할이 다르다.
    - `character_set_system`: 시스템 내부 기본 문자 집합
    - `character_set_server`: 서버 기본 문자 집합
    - `character_set_database`: DB 기본 문자 집합
    - `character_set_filesystem`: 파일명 해석용 문자 집합
    - `character_set_client`: 클라이언트가 보내는 SQL 문장의 문자 집합
    - `character_set_connection`: 클라이언트 입력값을 내부 처리용으로 변환할 때의 문자 집합
    - `character_set_results`: 결과를 클라이언트에 돌려줄 때의 문자 집합
        
        <img width="679" height="497" alt="image" src="https://github.com/user-attachments/assets/bcb29f28-e65f-43c6-833b-54c79bb825bb" />

- 15.1.3.1 클라이언트로부터 쿼리를 요청했을 때의 문자 집합 변환
    - 서버는 클라이언트에서 전달된 SQL 문장을 `character_set_client` 기준으로 해석하고, 필요하면 `character_set_connection` 기준으로 변환해 처리한다.
    - 리터럴 앞에 `_latin1`, `_utf8mb4` 같은 introducer를 붙이면 해당 문자열 상수의 문자 집합을 명시할 수 있다.
    - 이 값은 칼럼 문자 집합과 다를 수 있으며, 비교 시 변환 규칙의 영향을 받는다.
- 15.1.3.2 처리 결과를 클라이언트로 전송할 때의 문자 집합 변환
    - 실행 결과는 `character_set_results` 기준으로 변환되어 반환된다.
    - `SET NAMES utf8mb4`는 `character_set_client`, `character_set_connection`, `character_set_results`를 한 번에 맞추는 효과가 있다.
    - `CHARSET utf8mb4`도 비슷한 목적이지만 적용 범위와 동작 타이밍에 차이가 있다.
    - 클라이언트와 서버의 문자 집합이 어긋나면 저장은 정상인데 화면 출력만 깨지는 문제도 생길 수 있다.

### 15.1.4 콜레이션(Collation)

- 콜레이션은 문자열의 비교와 정렬 규칙이다.
- 같은 문자 집합이라도 콜레이션이 다르면 대소문자 구분 여부, 악센트 구분 여부, 정렬 순서가 달라진다.
- 콜레이션 이름은 보통 문자 집합명과 성격을 함께 담는다.
    - `_ci`: 대소문자 비구분
    - `_cs`: 대소문자 구분
    - `_bin`: 바이너리 비교
- utf8mb4 계열에서는 UCA 버전에 따른 차이도 중요하다.
- `utf8mb4_general_ci`는 오래된 단순 규칙 기반이고, `utf8mb4_0900_ai_ci`는 UCA 9.0 기준을 반영한다.
- `ai`는 악센트 비구분, `as`는 악센트 구분을 의미한다.
- 대소문자만이 아니라 악센트와 문자 정규화까지 비교 결과에 영향을 준다.
- MySQL 8.0에서는 기본 콜레이션이 `utf8mb4_0900_ai_ci` 계열로 바뀌었다.
- MySQL 5.7에서 8.0으로 업그레이드하면 기존 객체와 신규 객체의 기본 콜레이션이 달라질 수 있으므로 조인, 비교, 정렬 시 오류나 혼합 문제가 발생할 수 있다.
- 연결 파라미터에 `connectionCollation` 같은 설정을 추가해 애플리케이션 연결 시점의 콜레이션을 맞추는 것이 중요하다.
- 15.1.4.1 콜레이션 이해
    - 같은 칼럼 타입이라도 콜레이션에 따라 `a`, `A`, `á` 등의 비교 결과와 정렬 결과가 달라진다.
    - 테이블, 칼럼, 리터럴이 서로 다른 콜레이션을 가지면 비교 시 변환 규칙이 추가된다.
    - 비교와 정렬이 많은 칼럼은 문자 집합만이 아니라 콜레이션까지 설계 대상이다.
    - `information_schema.columns`에서 칼럼별 문자 집합과 콜레이션을 확인할 수 있다.
- 15.1.4.2 utf8mb4 문자 집합의 콜레이션
    - `utf8mb4_general_ci`는 단순하고 빠르지만 정교함은 떨어진다.
    - `utf8mb4_0900_ai_ci`는 더 정교한 비교 규칙을 제공하지만 상대적으로 무겁다.
    - 성능만 보고 무조건 `general_ci`를 선택할 것이 아니라, 검색 정확도와 정렬 규칙 요구사항을 같이 봐야 한다.
    - 서비스에서 국제화 요구가 크면 최신 UCA 기반 콜레이션이 더 적합할 수 있다.

### 15.1.5 비교 방식

- MySQL의 문자열 비교는 문자 집합과 콜레이션의 영향을 직접 받는다.
- CHAR 타입에서는 끝 공백 처리도 비교에 영향을 준다.
- `'ABC' = 'ABC '` 비교는 공백 무시 규칙 때문에 참이 될 수 있다.
- 하지만 `LIKE` 비교는 일반 비교와 다르게 동작할 수 있으며, 공백을 더 민감하게 다룬다.
- 동일 문자열 비교, 정렬, LIKE 검색의 결과가 모두 같을 것이라고 가정하면 안 된다.
- 콜레이션이 섞이면 `Illegal mix of collations` 오류가 날 수 있다.
- 특히 업그레이드 과정에서 기존 테이블은 옛 콜레이션, 신규 연결은 새 콜레이션을 쓰는 상황을 주의해야 한다.

### 15.1.6 문자열 이스케이프 처리

- MySQL은 문자열 안의 특수 문자를 `\`로 이스케이프할 수 있다.
- 대표적으로 `\0`, `\'`, `\"`, `\b`, `\n`, `\r`, `\t`, `\\`, `\%`, `\_` 등이 있다.
- `LIKE` 검색에서는 `%`와 `_`가 패턴 문자이므로 실제 문자로 검색하려면 이스케이프가 필요하다.
- 홑따옴표와 쌍따옴표는 SQL 모드와 문자열 표기 방식에 따라 해석이 달라질 수 있다.
- 백슬래시를 포함한 문자열은 클라이언트, SQL 모드, 애플리케이션 언어의 문자열 리터럴 규칙이 함께 얽히므로 주의해야 한다.
- 문자열 검색 오류나 예상과 다른 매칭이 생기면 실제 저장값과 이스케이프 규칙을 함께 확인해야 한다.

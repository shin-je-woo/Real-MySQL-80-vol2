# 01. 쿼리 작성 및 최적화

- 애플리케이션에서 데이터를 저장 또는 조회하기 위해 데이터베이스 서버로 전달하는 것은 SQL뿐이다.
- SQL은 어떠한(What) 데이터를 요청하기 위한 언어이지, 어떻게(How) 데이터를 읽을지를 표현하는 언어는 아니다.
- 그래서 SQL을 작성하는 방법이나 규칙은 물론, 내부적인 처리 방식(옵티마이저)에 대해 어느 정도의 지식이 필요하다.

## 11.1 쿼리 작성과 연관된 시스템 변수

### 11.1.1 SQL 모드

- `sql_mode`는 SQL 문장 작성 규칙뿐 아니라 데이터 타입 변환, 기본값 제어 같은 서버 내부 동작에도 영향을 준다.
- 운영 중인 MySQL 서버에서 `sql_mode`를 바꾸면 기존 애플리케이션 동작이 달라질 수 있으므로, 가능하면 기본값을 유지하거나 영향 범위를 충분히 검토한 뒤 적용하는 편이 안전하다.
- MySQL 8.0 기본 `sql_mode`는 다음 옵션 조합으로 제시된다.
    - `ONLY_FULL_GROUP_BY`
    - `STRICT_TRANS_TABLES`
    - `NO_ZERO_IN_DATE`
    - `NO_ZERO_DATE`
    - `ERROR_FOR_DIVISION_BY_ZERO`
    - `NO_ENGINE_SUBSTITUTION`
- MySQL 8.0으로 업그레이드할 때 기존 버전의 `sql_mode`를 유지하면 호환성 측면에서 유리할 수 있다. 별도 설정 없이 업그레이드하면 `ONLY_FULL_GROUP_BY` 영향에 특히 주의해야 한다.
- `STRICT_ALL_TABLES`, `STRICT_TRANS_TABLES`
    - `INSERT`, `UPDATE` 시 칼럼 타입과 저장 값 타입이 다를 때 자동 변환을 허용할지, 경고나 에러를 발생시킬지 결정한다.
    - `STRICT_TRANS_TABLES`는 트랜잭션 지원 스토리지 엔진에만 엄격 모드를 적용한다.
    - `STRICT_ALL_TABLES`는 모든 스토리지 엔진에 엄격 모드를 적용한다.
    - 값 자동 보정이 숨어 있는 데이터 오류로 이어질 수 있으므로, 서비스 적용 전에는 기존 입력 SQL이 암묵적 타입 변환에 의존하지 않는지 점검해야 한다.
- `ANSI_QUOTES`
    - 쌍따옴표를 문자열 표기가 아니라 식별자 구분 용도로 사용하게 만든다.
    - 이 옵션이 활성화되면 문자열은 홑따옴표만 사용해야 한다.
- `ONLY_FULL_GROUP_BY`
    - `GROUP BY` 절에 명시되지 않은 칼럼을 집계 함수 없이 `SELECT`, `HAVING`에 사용하는 것을 제한한다.
    - MySQL 5.7부터 기본값에 포함되고, MySQL 8.0에서도 기본적으로 적용된다.
    - 업그레이드 시 기존의 느슨한 `GROUP BY` 쿼리가 실패할 수 있다.
- `PIPES_AS_CONCAT`
    - `||`를 OR가 아니라 문자열 연결 연산자로 사용하게 만든다.
- `PAD_CHAR_TO_FULL_LENGTH`
    - `CHAR` 타입 값 조회 시 뒤쪽 공백 처리 방식에 영향을 준다.
- `NO_BACKSLASH_ESCAPES`
    - 역슬래시를 이스케이프 문자로 사용할 수 없게 만든다.
- `IGNORE_SPACE`
    - 함수명과 괄호 사이 공백을 허용한다.
    - 대신 내장 함수명이 예약어처럼 동작할 수 있어 식별자 이름 선택에 영향을 줄 수 있다.
- `REAL_AS_FLOAT`
    - `REAL`을 `DOUBLE`이 아니라 `FLOAT` 의미로 사용하게 만든다.
- `NO_ZERO_IN_DATE`, `NO_ZERO_DATE`
    - `2020-00-00`, `0000-00-00` 같은 잘못된 날짜 저장을 허용하지 않는다.
- `ANSI`
    - 여러 옵션을 묶어서 ANSI SQL 표준에 가깝게 동작하게 만드는 조합 모드다.
- `TRADITIONAL`
    - `STRICT_TRANS_TABLES`, `STRICT_ALL_TABLES`, `NO_ZERO_IN_DATE`, `NO_ZERO_DATE`, `ERROR_FOR_DIVISION_BY_ZERO`, `NO_ENGINE_SUBSTITUTION` 등을 묶은 더 엄격한 조합 모드다.
    - 기존에는 경고로 넘어가던 상황도 에러가 되어 SQL 문장이 실패할 수 있다.

### 11.1.2 영문 대소문자 구분

- MySQL은 설치된 운영체제에 따라 테이블명의 대소문자 구분 여부가 달라질 수 있다.
- 이는 DB나 테이블이 디스크의 디렉터리와 파일에 매핑되기 때문이다.
- `lower_case_table_names` 시스템 변수로 동작을 제어할 수 있다.
    - `1`이면 모두 소문자로 저장되고, MySQL 서버도 대소문자를 구분하지 않는다.
    - 윈도우와 macOS에서는 `2`를 사용할 수 있으며, 저장은 대소문자를 유지하지만 비교는 대소문자를 구분하지 않는다.
    - 기본값은 `0`이며, 이 경우 DB와 테이블명에서 대소문자를 구분한다.
- 설정과 별개로, 초기부터 DB명과 테이블명을 한 가지 규칙으로 통일하는 편이 안전하다.

### 11.1.3 MySQL 예약어

- 데이터베이스, 테이블, 칼럼 이름을 예약어와 같은 키워드로 만들면 SQL에서 사용할 때 항상 역따옴표로 감싸야 한다.
- 예약어 수가 적지 않고, 예약어처럼 보이지만 실제로는 문제가 되지 않는 키워드도 있어서 모두 외우기 어렵다.
- 예약어 여부가 헷갈릴 때는 역따옴표 없이 직접 테이블 생성 시도를 해보는 방법이 가장 확실하다.
    - 역따옴표 없이 생성이 되면 예약어가 아니다.
    - 생성이 실패하면 예약어이므로 역따옴표 없이 사용할 수 없다.
- 생성 시 역따옴표로 감쌌다고 해도 장기적으로는 좋지 않다. 처음부터 예약어를 피한 이름을 쓰는 편이 낫다.

## 11.2 매뉴얼의 SQL 문법 표기를 읽는 방법

- MySQL 매뉴얼의 SQL 문법 표기는 사용할 수 있는 모든 키워드와 기능을 한 문장에 압축해 표현한다.
- 문법 표기를 읽는 법을 알아야 매뉴얼을 정확히 해석할 수 있다.

- 대문자로 표기된 단어는 모두 키워드를 뜻한다.
- 이탤릭체는 사용자가 선택해서 작성하는 토큰을 뜻한다. 보통 테이블명, 칼럼명, 표현식 같은 값이다.
- 대괄호 `[]`
    - 해당 키워드나 표현식이 선택 사항임을 뜻한다.
- 파이프 `|`
    - 앞뒤 키워드나 표현식 중 하나만 선택할 수 있음을 뜻한다.
- 중괄호 `{}`
    - 괄호 안의 항목 중 반드시 하나를 사용해야 함을 뜻한다.
- `...`
    - 앞에 나온 키워드나 표현식 조합이 반복될 수 있음을 뜻한다.
- SQL 문장은 문법 표기에 제시된 순서를 따라 작성해야 한다.

## 11.3 MySQL 연산자와 내장 함수

- MySQL은 다른 DBMS와 비슷한 기본 연산자를 제공하지만, MySQL에서만 허용되거나 동작 방식이 다른 연산자와 함수도 많다.
- 가능하면 SQL 가독성을 높이기 위해 ANSI 표준 형태의 연산자와 문법 사용을 권장한다.
- 내장 함수는 DBMS마다 이름과 사용법 호환성이 낮으므로, MySQL 고유 함수는 목적과 제약을 분명히 알고 써야 한다.

### 11.3.1 리터럴 표기법 문자열

- 문자열, 숫자, 날짜, 불리언은 MySQL이 자동 타입 변환을 많이 수행하는 영역이다.
- 편리하지만, 자동 변환 때문에 성능 저하나 예기치 않은 비교 결과가 발생할 수 있다.
- 문자열
    - SQL 표준에서는 문자열을 홑따옴표로 감싼다.
    - MySQL은 쌍따옴표도 문자열 표기에 사용할 수 있지만, `ANSI_QUOTES`가 활성화되면 쌍따옴표는 식별자 표기 용도로만 사용된다.
    - 문자열 안에 홑따옴표가 포함되면 홑따옴표를 두 번 연속 써서 표현할 수 있다.
    - 식별자와 예약어 충돌을 피하려면 MySQL에서는 역따옴표를 사용한다.
    - 문자열은 홑따옴표, 식별자는 역따옴표로 역할을 분리하는 편이 안전하다.
- 숫자
    - 숫자 상수는 따옴표 없이 쓰는 것이 일반적이지만, MySQL은 문자열과 숫자를 비교할 때 자동 타입 변환을 수행한다.
    - 숫자 타입과 문자열 타입을 비교하면 MySQL은 숫자 타입을 우선하므로 문자열 값을 숫자로 바꿔 비교한다.
    - 문자열 칼럼을 숫자 상수와 비교하면 칼럼 전체 값이 숫자로 변환될 수 있어 인덱스를 제대로 쓰지 못할 수 있다.
    - 변환 불가능한 문자열이 섞여 있으면 쿼리가 실패할 수 있다.
    - 숫자 성격의 값은 가능하면 숫자 타입 칼럼에 저장해야 한다.
- 날짜
    - MySQL은 정해진 형식의 날짜 문자열을 자동으로 `DATE`, `DATETIME`으로 변환해 비교할 수 있다.
    - `STR_TO_DATE()`를 쓰지 않아도 되는 경우가 많다.
    - 문자열과 날짜 칼럼 비교에서도 날짜 칼럼의 인덱스 사용에는 문제가 없다.
    - 형식이 명확하지 않은 문자열이라면 명시적 변환이 더 안전하다.
- 불리언
    - `BOOL`, `BOOLEAN`은 독립적인 타입이 아니라 `TINYINT`의 동의어다.
    - `TRUE`, `FALSE`는 내부적으로 1과 0처럼 취급된다.
    - MySQL은 C/C++ 언어처럼 불리언 값을 정수로 매핑해 사용한다.
    - 그러나 모든 숫자가 깔끔하게 `TRUE`, `FALSE` 둘 중 하나로 관리되는 것은 아니므로 혼란과 버그 가능성이 있다.
    - 불리언 의미를 엄격하게 유지하려면 별도 제약이나 `ENUM` 관리 방식을 고려할 수 있다.

### 11.3.2 MySQL 연산자

- 동등(Equal) 비교 (`=`, `<=>`)
    - 일반적인 동등 비교는 `=`를 사용한다.
    - `NULL`은 `IS NULL` 없이 비교가 어렵고, `=`로 비교하면 한쪽이 `NULL`일 때 결과도 `NULL`이 된다.
    - MySQL은 Null-Safe 비교 연산자 `<=>`를 제공한다.
    - `<=>`는 양쪽이 모두 `NULL`이면 참, 한쪽만 `NULL`이면 거짓을 반환한다.
    - `NULL`을 하나의 비교 가능한 값처럼 다뤄야 할 때 유용하다.
- 부정(Not-Equal) 비교 (`<>`, `!=`)
    - 같지 않다 비교에는 일반적으로 `<>`를 많이 사용한다.
    - MySQL에서는 `!=`도 같은 의미로 사용할 수 있다.
    - 둘 다 동작하지만, 한 SQL 문장 안에서는 한 가지 방식으로 통일하는 편이 가독성에 좋다.
- NOT 연산자 (`!`)
    - 부정에는 `NOT`을 사용할 수 있고, C/C++처럼 `!`도 사용할 수 있다.
    - `!1`은 0, `!FALSE`는 1로 평가된다.
    - 숫자나 문자열 표현식에도 사용할 수 있지만, 결과 예측이 명확하지 않은 경우에는 남용하지 않는 편이 좋다.
- AND(`&&`)와 OR(`||`) 연산자
    - MySQL은 `AND`, `OR`뿐 아니라 `&&`, `||`도 허용한다.
    - `&&`는 `AND`와 같고, `||`는 기본적으로 `OR`와 같다.
    - 단 `PIPES_AS_CONCAT`가 활성화되면 `||`는 문자열 연결 연산자가 된다.
    - AND와 OR를 함께 쓸 때는 우선순위를 정확히 알아야 한다.
    - [주의] `AND`가 `OR`보다 먼저 처리된다.
    - 우선순위 오해를 줄이려면 괄호를 명시적으로 쓰는 편이 안전하다.
- 나누기(`/`, `DIV`)와 나머지(`%`, `MOD`) 연산자
    - `/`는 일반 나눗셈이다.
    - `DIV`는 나눗셈 결과의 정수 부분만 반환한다.
    - `%`와 `MOD()`는 나머지를 반환한다.
    - 같은 계산이라도 어떤 결과가 필요한지에 따라 연산자를 구분해서 써야 한다.
- `REGEXP` 연산자
    - 문자열이 특정 패턴을 만족하는지 확인하는 정규 표현식 비교 연산자다.
    - `RLIKE`는 `REGEXP`와 같은 역할을 한다.
    - 좌측에는 비교 대상 문자열이나 문자열 칼럼, 우측에는 정규 표현식을 둔다.
    - `^`, `$`, `[]`, `()`, `|`, `.`, , `+`, `?` 같은 정규 표현식 기호를 사용할 수 있다.
    - 정규 표현식은 강력하지만, 문자열 칼럼 비교에 사용할 때 인덱스 레인지 스캔을 사용할 수 없다.
    - 따라서 `WHERE REGEXP ...` 조건을 단독으로 쓰는 것은 성능상 좋지 않으며, 데이터 범위를 줄일 수 있는 다른 조건과 함께 쓰는 편이 낫다.
- `LIKE` 연산자
    - 정규 표현식보다 단순한 문자열 패턴 비교 연산자다.
    - `%`는 0개 이상 임의 문자열, `_`는 정확히 1개 문자를 뜻한다.
    - `ESCAPE` 절을 사용하면 `%`, `_`를 일반 문자로 비교할 수 있다.
    - `LIKE 'abc%'`처럼 접두어 패턴은 인덱스 레인지 스캔을 사용할 수 있다.
    - `LIKE '%rist'`처럼 앞부분이 와일드카드이면 인덱스의 Left-most 특성 때문에 인덱스 레인지 스캔을 쓸 수 없다.
    - 접두어 검색과 전체 포함 검색은 성능 차이가 매우 크다.
- `BETWEEN` 연산자
    - 크거나 같고, 작거나 같다는 두 비교를 하나로 묶은 연산자다.
    - 복합 인덱스에서 선행 칼럼에 `BETWEEN`이 걸리면 뒤 칼럼 조건이 범위 축소에 충분히 활용되지 못할 수 있다.
    - `(dept_no, emp_no)` 인덱스 예시에서 `dept_no BETWEEN 'd003' AND 'd005' AND emp_no=10001`은 넓은 범위를 읽는다.
    - 범위 비교라는 성격 때문에, 동등 비교 여러 개를 묶은 `IN`과는 실행 방식이 다르다.

- `IN` 연산자
    - 여러 값에 대해 동등 비교를 반복 수행하는 연산자다.
    - 상수 목록이 주어진 경우 `IN (?, ?, ?)` 형태는 빠르게 처리될 수 있다.
    - 복합 인덱스 예시에서는 `BETWEEN`보다 `IN`이 더 적은 레코드를 읽으며 효율적이다.
    - `(dept_no, emp_no)` 인덱스에서 `dept_no IN ('d003','d004','d005') AND emp_no=10001`은 `BETWEEN`보다 훨씬 좁은 범위만 읽는다.
    - MySQL 8.0 이전에는 `IN` 절에 튜플이 사용되면 항상 풀 테이블 스캔하는 문제가 있었지만, 8.0부터는 튜플을 그대로 나열해도 인덱스를 최적으로 사용할 수 있게 개선되었다.
    - `IN (subquery)` 형태도 MySQL 8.0에서는 세미 조인 최적화가 많이 안정화되었다.
    - 반면 `NOT IN`은 부정형 비교라 인덱스를 이용해 처리 범위를 줄이는 조건으로 쓰기 어렵다.

### 11.3.3 MySQL 내장 함수

- MySQL 함수는 서버가 기본 제공하는 내장 함수와 사용자가 작성해 추가하는 UDF(User Defined Function)로 구분된다.
- 여기서는 기본 제공 내장 함수 중 자주 쓰이는 것들이 정리된다.
- NULL 값 비교 및 대체 (`IFNULL`, `ISNULL`)
    - `IFNULL(expr1, expr2)`는 첫 번째 인자가 `NULL`이면 두 번째 인자를 반환한다.
    - 첫 번째 인자가 `NULL`이 아니면 첫 번째 인자를 그대로 반환한다.
    - `ISNULL(expr)`는 인자가 `NULL`이면 1, 아니면 0을 반환한다.
    - 조회 결과 기본값 대체나 간단한 `NULL` 검사에 자주 사용된다.
- 현재 시각 조회 (`NOW`, `SYSDATE`)
    - 두 함수 모두 현재 시각을 반환하지만 동작 방식이 다르다.
    - 하나의 SQL 문장 안에서는 `NOW()`가 같은 값을 유지한다.
    - `SYSDATE()`는 호출 시점마다 다시 평가되어 같은 SQL 안에서도 값이 달라질 수 있다.
    - `SLEEP()`과 함께 실행한 예시에서 `NOW()`는 동일값, `SYSDATE()`는 대기 시간만큼 차이 나는 값이 반환된다.
    - `SYSDATE()`는 레플리카 복제 안정성에 문제를 만들 수 있다.
    - `SYSDATE()`와 비교되는 칼럼은 인덱스를 효율적으로 사용하지 못할 수 있다.
    - 특별한 이유가 없으면 `NOW()`를 사용하는 편이 낫다.
    - 이미 `SYSDATE()`를 쓰고 있다면 `sysdate-is-now` 설정으로 `NOW()`처럼 동작시킬 수 있다.
- 날짜와 시간의 포맷 (`DATE_FORMAT`, `STR_TO_DATE`)
    - `DATE_FORMAT()`은 날짜나 시간을 원하는 형식의 문자열로 바꾼다.
    - `%Y`, `%m`, `%d`, `%H`, `%i`, `%s` 같은 지정자를 사용한다.
    - `STR_TO_DATE()`는 문자열을 지정한 포맷에 맞춰 `DATE`, `DATETIME`으로 변환한다.
    - 외부 입력 문자열을 날짜 타입으로 바꿔야 할 때 유용하다.
- 날짜와 시간의 연산 (`DATE_ADD`, `DATE_SUB`)
    - 날짜나 시간에 값을 더하거나 뺄 때 사용한다.
    - 두 함수 모두 `INTERVAL n unit` 형식의 두 번째 인자를 받는다.
    - `YEAR`, `MONTH`, `DAY`, `HOUR`, `MINUTE`, `SECOND`, `MICROSECOND`, `QUARTER`, `WEEK` 단위를 사용할 수 있다.
        
        <img width="462" height="339" alt="image" src="https://github.com/user-attachments/assets/aebf41c2-e891-4fe0-9714-38ecc5e10f48" />

- 타임스탬프 연산 (`UNIX_TIMESTAMP`, `FROM_UNIXTIME`)
    - `UNIX_TIMESTAMP()`는 `1970-01-01 00:00:00` 이후 경과 초를 반환한다.
    - 인자를 넘기면 해당 날짜/시간의 타임스탬프를 반환한다.
    - `FROM_UNIXTIME()`은 타임스탬프 값을 `DATETIME`으로 변환한다.
    - MySQL의 `TIMESTAMP` 타입은 4바이트 정수로 저장되므로 표현 가능한 날짜 범위가 제한된다.
- 문자열 처리 (`RPAD`, `LPAD`, `RTRIM`, `LTRIM`, `TRIM`)
    - `RPAD`, `LPAD`는 문자열 좌우에 문자를 덧붙여 지정 길이로 맞춘다.
    - `RTRIM`, `LTRIM`은 문자열 오른쪽, 왼쪽의 연속 공백 문자를 제거한다.
    - `TRIM`은 양쪽 공백을 제거한다.
- 문자열 결합 (`CONCAT`)
    - 여러 값을 문자열 하나로 이어 붙인다.
    - 숫자 인자를 넘기면 자동으로 문자열로 변환해 연결한다.
    - 의도와 다를 수 있으므로 필요하면 `CAST()`로 명시적 변환을 거치는 편이 안전하다.
    - `CONCAT_WS()`는 첫 번째 인자를 구분자로 사용한다.
- GROUP BY 문자열 결합 (`GROUP_CONCAT`)
    - 여러 레코드 값을 하나의 문자열로 결합하는 그룹 함수다.
    - `SEPARATOR`로 구분자를 바꿀 수 있다.
    - 함수 내부 `ORDER BY`로 결합 순서를 제어할 수 있다.
    - `DISTINCT`를 주면 중복 값을 제거할 수 있다.
    - 기본적으로 제한된 메모리 버퍼를 사용하며, 지정 크기를 초과하면 경고가 발생한다.
    - JDBC에서는 경고가 아니라 오류처럼 다뤄져 쿼리가 실패할 수 있어 주의해야 한다.
    - 버퍼 크기는 `group_concat_max_len` 시스템 변수로 조절한다.
    - MySQL 8.0 이전에는 그룹별 개수 제한이 쉽지 않았지만, 8.0부터는 윈도 함수와 래터럴 조인을 활용할 수 있다.
- 값의 비교와 대체 (`CASE WHEN ... THEN ... END`)
    - 함수가 아니라 SQL 구문이지만, 조건에 따라 값을 바꾸거나 분류할 때 자주 사용된다.
    - `CASE`로 시작하고 `END`로 끝난다.
    - 두 가지 형태가 있다.
        - 기준값과 `WHEN` 값을 비교하는 방식
        - `WHEN` 뒤에 비교식을 직접 쓰는 방식
    - `CASE WHEN` 절이 일치하는 경우에만 `THEN` 이하 표현식이 실행된다는 점이 중요하다.
    - 이 특성 때문에 필요할 때만 서브쿼리를 실행하도록 만들 수 있다.
- 타입의 변환 (`CAST`, `CONVERT`)
    - MySQL은 문자열, 숫자, 날짜를 자동 변환하기도 하지만, 명시적 변환이 필요할 때 `CAST()`를 사용한다.
    - `CAST()`로 `DATE`, `TIME`, `DATETIME`, `BINARY`, `CHAR`, `DECIMAL`, `SIGNED INTEGER`, `UNSIGNED INTEGER` 등으로 변환할 수 있다.
    - 부호 있는 정수를 `UNSIGNED`로 바꾸는 경우처럼 자동 변환과 기대 결과가 다를 수 있는 상황에서 명시적 타입 변환이 필요하다.
    - `CONVERT()`도 타입 변환에 사용할 수 있고, 문자 집합 변환에도 사용된다.
    - 문자 집합 변환은 `USING` 키워드를 사용한다.
- 이진값과 16진수 문자열 변환 (`HEX`, `UNHEX`)
    - `HEX()`는 이진값을 사람이 읽을 수 있는 16진수 문자열로 바꾼다.
    - `UNHEX()`는 16진수 문자열을 이진값으로 바꾼다.
    - 해시 결과를 `BINARY`, `VARBINARY`에 저장할 때 함께 사용된다.
- 암호화 및 해시 함수 (`MD5`, `SHA`, `SHA2`)
    - `MD5`, `SHA`, `SHA2`는 해시 값을 반환하는 함수다.
    - 결과는 16진수 문자열 형태여서 저장 시 원본 바이트 수보다 더 큰 공간이 필요하다.
    - 저장 공간을 줄이려면 `CHAR`, `VARCHAR` 대신 `BINARY`, `VARBINARY`에 `UNHEX()` 결과를 저장하는 방식이 유리하다.
    - 긴 문자열 전체에 직접 인덱스를 만들기 어렵거나 비효율적일 때, 해시 값을 별도 칼럼 또는 함수 기반 인덱스로 저장해 검색 성능을 개선할 수 있다.
    - MySQL 8.0부터는 함수 기반 인덱스를 지원하므로 별도 해시 칼럼 없이 `MD5(access_url)` 같은 표현식에 인덱스를 만들 수 있다.
    - 해시 검색에서는 평문과 해시 결과를 직접 비교하면 안 되고, 같은 해시 함수 결과끼리 비교해야 한다.
    - 해시 결과를 `UNHEX()`로 저장하면 저장 공간과 인덱스 크기를 더 줄일 수 있다.
- 처리 대기 (`SLEEP`)
    - 지정한 초만큼 대기하는 함수다.
    - 반환값보다 대기 시간 자체가 의미가 있다.
    - 개발, 디버깅, 테스트 용도로 유용하다.
    - 레코드 수만큼 호출되므로 `SELECT SLEEP(1.5) FROM ...` 같은 형태는 전체 실행 시간이 크게 늘 수 있다.
- 벤치마크 (`BENCHMARK`)
    - 어떤 표현식을 지정 횟수만큼 반복 실행해 성능을 대략 비교하는 함수다.
    - 반환값 자체보다는 반복 실행에 걸린 시간이 중요하다.
    - 함수나 표현식 간 상대 비교 용도로 쓰는 것이 적절하다.
    - `SELECT BENCHMARK(10, expr)`와 `SELECT expr`를 10번 직접 실행하는 것은 파싱, 최적화, 네트워크 비용이 달라 결과가 같지 않다.
- IP 주소 변환 (`INET_ATON`, `INET_NTOA`, `INET6_ATON`, `INET6_NTOA`)
    - IPv4 주소를 문자열이 아니라 부호 없는 정수 또는 바이너리 값으로 저장할 수 있게 변환한다.
    - `INET_ATON()`은 IPv4 문자열을 정수로 바꾸고, `INET_NTOA()`는 정수를 사람이 읽는 IPv4 문자열로 바꾼다.
    - `INET6_ATON()`, `INET6_NTOA()`는 IPv6도 처리할 수 있다.
    - IPv4는 `BINARY(4)`, IPv6는 `BINARY(16)` 저장이 가능하다.
    - IPv4와 IPv6를 함께 저장하려면 `VARBINARY(16)` 사용을 권장한다.
- JSON 포맷 (`JSON_PRETTY`)
    - JSON 칼럼을 읽기 쉬운 형태로 포맷해 준다.
    - 클라이언트 기본 출력은 단순 텍스트라 가독성이 떨어질 수 있다.
- JSON 필드 크기 (`JSON_STORAGE_SIZE`)
    - MySQL은 JSON을 디스크에 BSON 형식으로 저장한다.
    - `JSON_STORAGE_SIZE()`는 실제 저장 공간 크기를 바이트 단위로 반환한다.
    - `JSON_STORAGE_FREE()`와 함께 자주 사용될 수 있다.
- JSON 필드 추출 (`JSON_EXTRACT`)
    - JSON 문서에서 특정 경로의 값을 가져오는 가장 일반적인 함수다.
    - 첫 번째 인자는 JSON 문서, 두 번째 인자는 JSON Path다.
    - 결과는 따옴표를 포함한 JSON 값 형태다.
    - `JSON_UNQUOTE(JSON_EXTRACT(...))`를 사용하면 따옴표 없는 일반 문자열 값을 얻을 수 있다.
    - MySQL은 편의상 JSON 연산자도 제공한다.
        - `>` 는 `JSON_EXTRACT()`와 같은 역할이다.
        - `>>` 는 `JSON_UNQUOTE(JSON_EXTRACT())`와 같은 역할이다.
- JSON 오브젝트 포함 여부 확인 (`JSON_CONTAINS`)
    - 지정된 JSON 문서나 JSON 경로에 특정 JSON 값이 존재하는지 검사한다.
    - 첫 번째 인자는 JSON 문서, 두 번째 인자는 찾을 JSON 값이다.
    - 세 번째 인자로 JSON 경로를 지정하면 그 경로 안에서만 검사할 수 있다.
- JSON 오브젝트 생성 (`JSON_OBJECT`)
    - RDBMS 칼럼 값을 JSON 오브젝트 형태로 구성해 반환한다.
    - 키와 값을 번갈아 나열하는 방식으로 사용한다.
- JSON 칼럼으로 집계 (`JSON_OBJECTAGG`, `JSON_ARRAYAGG`)
    - `GROUP BY` 결과와 함께 사용해 여러 값을 JSON으로 집계하는 함수다.
    - `JSON_OBJECTAGG()`는 키-값 쌍의 JSON 문서를 만든다.
    - `JSON_ARRAYAGG()`는 값을 모아 JSON 배열을 만든다.
- JSON 데이터를 테이블로 변환 (`JSON_TABLE`)
    - JSON 문서 값을 행과 칼럼 구조로 풀어서 RDBMS 테이블처럼 반환하는 함수다.
    - `COLUMNS` 절에서 JSON Path와 칼럼 타입을 지정한다.
    - 반환되는 레코드 수는 원본 JSON 문서 수와 동일한 구조를 가진다.
    - 내부 임시 테이블을 사용하므로, 임시 테이블에 많은 레코드가 저장되지 않도록 주의해야 한다.
    - 실행 계획에서도 `Table function: json_table`, `Using temporary`가 표시된다.

## 11.4 SELECT

- SELECT는 온라인 트랜잭션 환경에서 가장 자주 사용되고, 여러 테이블과 많은 레코드를 빠르게 읽어야 하므로 성능 영향이 크다.
- 이 절은 SELECT 문 자체의 문법보다, 각 절이 어떤 순서로 처리되고 인덱스를 어떻게 타는지, 비교 조건과 정렬, 집계, 조인, 서브쿼리가 성능에 어떤 영향을 주는지를 중심으로 설명한다.

### 11.4.1 SELECT 절의 처리 순서

- SELECT 문은 보통 `SELECT`, `FROM`, `WHERE`, `GROUP BY`, `HAVING`, `ORDER BY`, `LIMIT` 절로 구성된다.
- 실제 실행 순서는 대부분 아래와 같다.

    - FROM과 JOIN으로 드라이빙 테이블과 드리븐 테이블을 읽는다.
    - WHERE 조건을 적용한다.
    - GROUP BY를 수행한다.
    - DISTINCT를 처리한다.
    - HAVING을 적용한다.
    - ORDER BY를 수행한다.
    - LIMIT를 적용한다.
- GROUP BY 없이 ORDER BY만 있는 경우처럼 예외적 실행 순서도 있다.

    - 먼저 정렬 가능한 테이블을 읽고 정렬한 뒤, 나머지 테이블을 조인하는 형태가 가능하다.
- 일반적인 절 처리 순서를 벗어나려면 인라인 뷰나 서브쿼리를 써서 처리 단계를 강제로 분리해야 한다.
- LIMIT을 GROUP BY보다 먼저 적용하고 싶다면 서브쿼리 안에서 LIMIT을 수행하고, 바깥 쿼리에서 GROUP BY와 ORDER BY를 해야 한다
- MySQL 8.0에서는 FROM 절의 서브쿼리를 외부 쿼리와 병합하는 최적화가 가능하다. 이 경우도 결국 SELECT의 일반 처리 순서 안에서 실행되는 형태가 된다.

### 11.4.2 WHERE 절과 GROUP BY 절, ORDER BY 절의 인덱스 사용

- 11.4.2.1 인덱스를 사용하기 위한 기본 규칙
    - WHERE, GROUP BY, ORDER BY가 인덱스를 타려면 인덱스 칼럼 값을 변형하지 않고 그대로 사용해야 한다.
    - 인덱스는 원본 값이 B-Tree에 정렬된 상태로 저장되므로, 칼럼에 연산이나 함수를 적용하면 인덱스 순서를 활용할 수 없다.
    - `salary*10 > 150000`처럼 칼럼에 연산을 거는 방식은 피하고, `salary > 150000/10`처럼 상수로 바꾸는 형태로 써야 한다.
    - MD5 같은 함수 결과 비교가 필요하면 가상 칼럼이나 함수 기반 인덱스를 사용해야 한다.
    - 비교 양쪽의 데이터 타입도 일치해야 한다.
    - 칼럼 타입과 상수 타입이 다르면 내부 타입 변환이 발생하고, 이때 인덱스 레인지 스캔이 불가능해질 수 있다.
- 11.4.2.2 WHERE 절의 인덱스 사용
    - WHERE 조건은 작업 범위 결정 조건과 체크 조건으로 나뉜다.
    - 작업 범위 결정 조건은 실제로 읽을 레코드 범위를 줄이는 조건이다.
    - 체크 조건은 읽어 온 레코드를 다시 거르는 조건이다.
    - 인덱스 선두 칼럼부터 동등 비교가 연속해서 맞으면 작업 범위 결정 조건으로 활용된다.
    - 중간에 범위 비교가 들어가거나 칼럼이 빠지면 그 뒤 칼럼은 주로 체크 조건으로만 쓰인다.
    - OR 조건이 나오면 처리 방식이 크게 달라진다.
        - AND는 읽을 레코드 수를 줄이는 방향이다.
        - OR는 여러 조건 결과를 합쳐야 해 읽을 레코드 수가 늘어난다.
        - 경우에 따라 index_merge가 가능하지만, 일반적으로 좋은 단일 인덱스 범위 스캔보다 느리다.
    - WHERE 절에 조건을 적는 순서 자체는 중요하지 않다.
- 11.4.2.3 GROUP BY 절의 인덱스 사용
    - GROUP BY는 비교 연산이 아니라 그룹핑이므로 WHERE처럼 작업 범위 결정 조건과 체크 조건으로 나누지 않는다.
    - GROUP BY 절의 칼럼 순서가 인덱스 칼럼 순서와 같아야 인덱스를 이용한 그룹핑이 가능하다.
        
        <img width="390" height="166" alt="image" src="https://github.com/user-attachments/assets/b273122c-5d62-4ee1-97b1-047bb0f7ecd9" />

    - 인덱스 앞부분 칼럼이 GROUP BY에서 빠지면 뒤 칼럼만으로는 인덱스를 이용하기 어렵다.
    - WHERE 절에서 인덱스 앞쪽 칼럼을 동등 비교로 고정했다면 GROUP BY에서 생략해도 인덱스를 계속 활용할 수 있는 경우가 있다.
    - 다음 유형은 인덱스를 이용한 GROUP BY가 가능하다.
        - `GROUP BY col_1`
        - `GROUP BY col_1, col_2`
        - `GROUP BY col_1, col_2, col_3`
    - 다음 유형은 인덱스를 이용하기 어렵다.
        - 인덱스 순서와 다른 순서로 GROUP BY 하는 경우
        - 중간 칼럼을 건너뛰는 경우
        - 인덱스에 없는 칼럼까지 GROUP BY 하는 경우
- 11.4.2.4 ORDER BY 절의 인덱스 사용
    - ORDER BY도 원리는 GROUP BY와 거의 같다.
    - ORDER BY 칼럼 순서가 인덱스 정의 순서와 같아야 한다.
        
        <img width="369" height="186" alt="image" src="https://github.com/user-attachments/assets/18977426-8ec2-477e-843b-9250f78e8c74" />

    - 정렬 방향까지 인덱스 정의와 맞아야 인덱스를 이용한 정렬이 가능하다.
    - MySQL 8.0 이전에는 ASC와 DESC를 혼합한 정렬에서 인덱스 활용이 어려웠다.
    - MySQL 8.0부터는 인덱스를 `col1 DESC, col2 ASC`처럼 정의할 수 있다.
    - 함수나 표현식 정렬은 MySQL 8.0부터 함수 기반 인덱스로 개선할 수 있다.
- 11.4.2.5 WHERE 조건과 ORDER BY 또는 GROUP BY 절의 인덱스 사용
    - WHERE와 ORDER BY가 함께 있을 때는 보통 다음 세 방식 중 하나다.
        - WHERE와 ORDER BY가 같은 인덱스를 함께 사용
        - WHERE만 인덱스를 사용하고 ORDER BY는 별도 정렬
        - ORDER BY만 인덱스를 사용하고 WHERE는 읽은 뒤 필터링
    - 가장 좋은 경우는 WHERE와 ORDER BY가 동시에 하나의 인덱스를 타는 경우다.
        
        <img width="464" height="224" alt="image" src="https://github.com/user-attachments/assets/8755d35c-81de-4f52-b040-ede6fbba0930" />

    - WHERE에서 동등 비교로 고정된 칼럼과 ORDER BY 칼럼이 인덱스 순서대로 이어져 있어야 한다.
    - WHERE가 범위 조건이 되면 ORDER BY는 인덱스를 못 타는 경우가 많다.
    - 중간 칼럼이 빠지거나 ORDER BY 순서가 어긋나면 WHERE와 ORDER BY를 동시에 인덱스로 처리할 수 없다.
    - WHERE와 GROUP BY 조합도 같은 기준으로 판단할 수 있다.
- 11.4.2.6 GROUP BY 절과 ORDER BY 절의 인덱스 사용
    - GROUP BY와 ORDER BY가 함께 있으면 둘 다 하나의 인덱스를 함께 써야 한다.
    - 둘 중 하나라도 인덱스를 못 타면, 보통 둘 다 인덱스를 못 탄다.
    - GROUP BY와 ORDER BY에 적은 칼럼의 순서와 내용이 같아야 한다.
    - MySQL 5.7까지는 GROUP BY가 정렬까지 같이 수행하는 경우가 많았지만, MySQL 8.0부터는 GROUP BY가 정렬을 보장하지 않는다.
    - MySQL 8.0에서는 그룹핑과 정렬이 둘 다 필요하면 GROUP BY와 ORDER BY를 모두 명시해야 한다.
- 11.4.2.7 WHERE 조건과 ORDER BY 절, GROUP BY 절의 인덱스 사용
    - 세 절이 함께 있을 때는 다음 순서로 판단한다.
        
        <img width="466" height="314" alt="image" src="https://github.com/user-attachments/assets/de717a47-393a-4f78-ad81-5ba936c25895" />

        - WHERE가 인덱스를 사용할 수 있는가
        - GROUP BY가 인덱스를 사용할 수 있는가
        - ORDER BY가 인덱스를 사용할 수 있는가
    - 셋 다 동시에 인덱스를 타는 경우가 가장 좋다.
    - WHERE만 인덱스를 타는 경우도 흔하다.
    - WHERE가 인덱스를 못 타면 GROUP BY와 ORDER BY의 인덱스 활용 여지도 크게 줄어든다.

### 11.4.3 WHERE 절의 비교 조건 사용 시 주의사항

- WHERE 비교 표현식은 인덱스 설계만큼 중요하다.
- 인덱스를 잘 만들어도 비교식을 잘못 쓰면 실행 계획이 나빠진다.
- 11.4.3.1 NULL 비교
    - MySQL에서는 NULL도 인덱스로 관리된다.
    - `NULL = NULL`은 참이 아니므로 NULL 비교는 `IS NULL` 또는 `<=>`를 써야 한다.
    - `ISNULL(col)` 함수도 쓸 수 있지만, WHERE에서 인덱스를 제대로 쓰지 못할 수 있다.
    - NULL 비교 목적이라면 가급적 `IS NULL`을 쓰는 편이 좋다.
    - `to_date IS NULL`은 인덱스를 타지만, `ISNULL(to_date)=1` 같은 형태는 인덱스를 못 탈 수 있다.
- 11.4.3.2 문자열이나 숫자 비교
    - 비교 대상 칼럼 타입에 맞는 상수를 써야 한다.
    - 숫자 칼럼에는 숫자 리터럴, 문자열 칼럼에는 문자열 리터럴을 사용하는 것이 원칙이다.
    - 숫자 칼럼에 문자열 상수를 쓰면 MySQL이 숫자로 바꿔 비교해 큰 문제 없이 지나갈 수 있다.
    - 반대로 문자열 칼럼에 숫자 상수를 쓰면 칼럼 값을 숫자로 바꾸려 해 인덱스를 못 탈 수 있다.
- 11.4.3.3 날짜 비교
    - 날짜 비교는 DATE, DATETIME, TIMESTAMP, 문자열이 섞이는 순간 주의가 필요하다.
    - DATE 또는 DATETIME과 문자열 비교
        - `hire_date > '2011-07-23'`처럼 문자열을 쓰면 MySQL이 내부적으로 DATE 또는 DATETIME으로 변환해 비교한다.
        - 반대로 칼럼 쪽에 `DATE_FORMAT(hire_date, ...)`, `DATE_ADD(hire_date, ...)` 같은 함수를 적용하면 인덱스를 못 탄다.
        - 날짜 비교는 칼럼이 아니라 상수를 변환하는 형태로 써야 한다.
    - DATE와 DATETIME의 비교
        - DATETIME과 DATE 비교는 내부 변환을 거쳐 수행된다.
        - 이때는 인덱스 사용 여부보다 결과 해석이 더 중요하다.
        - DATE와 DATETIME 비교는 직관과 다를 수 있어 결과를 꼭 확인해야 한다.
    - DATETIME과 TIMESTAMP의 비교
        - `UNIX_TIMESTAMP()` 결과는 숫자이므로 DATETIME과 직접 비교하면 원하는 결과가 나오지 않을 수 있다.
        - DATETIME과 비교하려면 `FROM_UNIXTIME(UNIX_TIMESTAMP())`처럼 DATETIME으로 맞춰 써야 한다.
        - 또는 `NOW()`를 쓰는 편이 단순하다.
- 11.4.3.4 Short-Circuit Evaluation
    - AND나 OR로 연결된 조건식은 앞 조건 결과에 따라 뒤 조건 평가를 생략할 수 있다.
    - 인덱스를 타지 못하는 조건끼리 연결된 경우에는 더 빨리 걸러지는 조건을 앞에 두는 편이 유리할 수 있다.
    - 예시에서는 `to_date<'1985-01-01'`을 먼저 두었을 때 `CONVERT_TZ(from_date, ...) > '1991-01-01'`보다 더 빨랐다.
    - 하지만 인덱스를 사용할 수 있는 조건이 있다면 MySQL은 short-circuit와 무관하게 그 조건을 우선 활용한다.
    - 즉 WHERE에 적은 순서가 항상 성능을 결정하지는 않는다.
    - EXISTS 서브쿼리 같은 복잡한 조건은 가능하면 뒤쪽에 두는 편이 낫다.
    - 실제로 가장 먼저 평가되는 것은 SQL에 먼저 적은 조건이 아니라 인덱스를 가장 잘 활용할 수 있는 조건이다.

### 11.4.4 DISTINCT

- DISTINCT는 특정 칼럼 조합의 중복을 제거할 때 사용한다.
- 조인으로 중복이 불어난 원인을 이해하지 못한 채 DISTINCT를 덮어쓰는 경우가 많다.
- DISTINCT는 결과 자체를 바꾸는 기능이므로, 성능뿐 아니라 의미도 바뀔 수 있다.
- 여러 테이블 조인에서 DISTINCT를 쓰기 전에 조인 관계가 맞는지 먼저 확인해야 한다.
- DISTINCT와 LIMIT이 같이 쓰이면 유니크한 값이 필요한 개수만큼 모이는 순간 처리를 멈출 수 있다.
- 정렬이 없거나 인덱스를 적절히 타는 경우에는 DISTINCT와 LIMIT 조합이 작업량을 줄여 주기도 한다.

### 11.4.5 LIMIT n

- LIMIT은 최종 결과에서 필요한 개수만 가져오고 나머지는 버릴 때 사용한다.
- LIMIT은 WHERE처럼 먼저 적용되는 것이 아니라 ORDER BY 이후 최종 결과에서 일부만 반환하는 역할이다.
- 정렬이나 그룹핑이 없다면 LIMIT만으로도 읽기 작업을 빨리 멈출 수 있다.
- 하지만 GROUP BY나 DISTINCT와 함께 쓰면 작업이 끝난 뒤에야 LIMIT이 적용되어 작업량 감소 효과가 작아질 수 있다.
- LIMIT은 인자 1개 또는 2개를 사용할 수 있다.
    - `LIMIT 10`은 상위 10건
    - `LIMIT 10, 10`은 11번째부터 10건
- `LIMIT 200000, 10`처럼 큰 오프셋은 매우 비효율적이다.
    - 앞의 200000건을 읽고 버린 뒤 마지막 10건만 반환하기 때문이다.
- 큰 페이지 번호가 자주 등장하면 조건 기반 페이징이 더 적합하다.
- 이전 페이지 마지막 값으로 `salary > 이전값 AND NOT (salary=이전값 AND emp_no<=이전값)` 같은 조건을 사용하면 중복과 누락을 줄일 수 있다.

### 11.4.6 COUNT()

- COUNT()는 결과 레코드 수를 반환한다.
- `COUNT(*)`의 는 모든 칼럼을 읽는 뜻이 아니라 레코드 자체를 센다는 뜻이다.
- 그래서 `COUNT(*)`, `COUNT(1)`, `COUNT(프라이머리 키)`는 일반적으로 같은 처리 성능을 보인다.
- `COUNT(expr)`는 expr 결과가 NULL이 아닌 레코드 수만 센다.
- NULL 가능한 칼럼에 `COUNT(col)`을 쓰면 의도와 다른 결과가 나올 수 있다.
- 게시판 목록처럼 전체 건수 COUNT와 목록 SELECT를 같이 호출하는 경우가 많은데, COUNT(*)가 오히려 더 느릴 수 있다.
- COUNT(*)에는 보통 ORDER BY가 필요 없고, 레코드 수 변화에 영향 없는 LEFT JOIN도 제거하는 편이 좋다.
- MySQL 8.0부터는 SELECT COUNT(*)에서 ORDER BY를 옵티마이저가 무시하도록 개선되었지만, 처음부터 불필요한 절을 빼는 편이 더 낫다.
- 실무에서는 전체 건수 대신 이전, 다음 링크만 보여 주는 식으로 COUNT 호출 자체를 줄이는 방법도 고려할 수 있다.

### 11.4.7 JOIN

- 11.4.7.1 JOIN의 순서와 인덱스
    - 조인에서 드라이빙 테이블은 먼저 읽는 테이블이고, 드리븐 테이블은 그 결과로 반복 조회되는 테이블이다.
    - 인덱스 탐색은 부하가 높고, 스캔은 상대적으로 낮으므로 드라이빙 테이블 선택이 전체 비용에 큰 영향을 준다.
    - 양쪽 조인 칼럼에 모두 인덱스가 있다면 어느 쪽을 드라이빙으로 잡아도 비교적 효율적이다.
    - 한쪽에만 인덱스가 있다면, 보통 인덱스 없는 쪽을 드라이빙으로 잡고 인덱스 있는 쪽을 드리븐으로 사용하는 편이 유리하다.
    - 양쪽 다 인덱스가 없으면 레코드 건수가 적은 테이블을 드라이빙으로 고르는 것이 일반적이다.
    - MySQL 8.0 이전에는 블록 네스티드 루프 조인이 주로 쓰였지만, 8.0.18부터는 해시 조인이 도입되었다.
- 11.4.7.2 JOIN 칼럼의 데이터 타입
    - 조인 칼럼은 양쪽 데이터 타입을 일치시켜야 한다.
    - 타입이 다르면 조인 비교를 위해 변환이 필요해지고, 이때 인덱스를 효율적으로 쓰지 못한다.
    - CHAR와 INT처럼 종류가 다른 경우가 대표적이다.
    - 같은 CHAR라도 문자 집합이나 콜레이션이 다르면 문제가 될 수 있다.
    - 이런 문제는 표준화된 데이터 모델과 일관된 타입 설계로 예방하는 것이 가장 좋다.
- 11.4.7.3 OUTER JOIN의 성능과 주의사항
    - OUTER JOIN은 INNER JOIN보다 조인 순서 최적화 여지가 적다.
    - LEFT JOIN에서는 아우터로 조인되는 테이블을 드라이빙 테이블로 선택할 수 없어 실행 계획이 불리해질 수 있다.
    - INNER JOIN으로 써도 되는 쿼리를 OUTER JOIN으로 쓰면 옵티마이저의 선택 폭을 줄여 성능이 나빠질 수 있다.
    - OUTER JOIN 대상 테이블의 조건을 WHERE에 두면 조인의 의미가 바뀐다.
    - 예를 들어 LEFT JOIN한 테이블 조건을 `WHERE mgr.dept_no='d001'`로 적으면 결국 INNER JOIN처럼 동작한다.
    - 정상적인 LEFT JOIN 의미를 유지하려면 그런 조건은 ON 절로 옮겨야 한다.
    - 예외적으로 안티 조인처럼 조인되지 않은 행만 찾고 싶을 때는 `LEFT JOIN ... WHERE joined_col IS NULL` 형태를 사용한다.
- 11.4.7.4 JOIN과 외래키
    - 외래키가 있어야만 JOIN할 수 있는 것은 아니다.
    - 외래키는 참조 무결성을 보장하기 위한 장치이고, JOIN은 값을 조건식으로 연결해 읽는 기능이다.
    - 외래키가 없어도 JOIN은 가능하다.
    - 다만 데이터 모델의 관계를 분명히 하고 무결성을 보장하려면 외래키가 유용하다.
- 11.4.7.5 지연된 조인(Delayed Join)
    - GROUP BY나 ORDER BY를 인덱스로 해결하지 못하면, MySQL은 보통 조인을 먼저 다 수행한 뒤 그룹핑이나 정렬을 한다.
    - 조인 결과 레코드 수가 크면 매우 비효율적이다.
    - 지연된 조인은 조인 전에 GROUP BY, ORDER BY, LIMIT을 먼저 수행해 대상 레코드 수를 줄인 뒤 마지막에 필요한 소수의 레코드만 조인하는 방식이다.
    - 특히 LIMIT이 있는 쿼리에서 효과가 크다.
    - 예시에서는 원래 쿼리가 salaries와 employees를 먼저 조인하고 그룹핑과 정렬 뒤 10건만 반환했다.
    - 지연된 조인으로 바꾸면 salaries에서 먼저 WHERE, GROUP BY, ORDER BY, LIMIT을 처리해 10건만 남긴 뒤 employees와 조인한다.
    - 하지만 모든 쿼리에 적용할 수 있는 것은 아니다.
        - LEFT JOIN은 드라이빙과 드리븐이 1:1 또는 M:1 관계여야 한다.
        - INNER JOIN은 드라이빙의 각 레코드가 드리븐에 반드시 존재하고, 서브쿼리에 LIMIT을 넣어도 최종 결과 개수가 바뀌지 않아야 한다.
    - 지연된 조인의 핵심은 조인 횟수를 줄이는 것이 아니라, GROUP BY와 ORDER BY가 처리해야 할 레코드 수를 줄이는 것이다.
- 11.4.7.6 래터럴 조인(Lateral Join)
    - MySQL 8.0 전에는 FROM 절 서브쿼리가 외부 쿼리 칼럼을 참조하기 어려웠다.
    - MySQL 8.0부터는 `LATERAL` 키워드로 FROM 절 서브쿼리가 외부 칼럼을 참조할 수 있다.
    - 예시에서는 특정 사원의 최근 급여 이력 2건만 가져오기 위해 `LEFT JOIN LATERAL (...)`을 사용한다.
    - LATERAL 없이 일반 서브쿼리로 외부 칼럼을 참조하면 에러가 난다.
    - 래터럴 조인은 외부 쿼리 각 레코드마다 임시 결과가 생성되는 구조라 꼭 필요할 때만 사용해야 한다.
    - 같은 레코드의 여러 칼럼을 얻기 위해 SELECT 절 서브쿼리를 여러 번 반복하는 대신, 래터럴 조인 하나로 묶을 수 있다.
- 11.4.7.7 실행 계획으로 인한 정렬 흐트러짐
    - MySQL 8.0 이전에는 네스티드 루프 조인만 가능해서 드라이빙 테이블을 읽은 순서가 결과 순서에 반영되는 경우가 많았다.
    - MySQL 8.0부터는 해시 조인이 도입되어 드라이빙 순서와 결과 순서가 달라질 수 있다.
    - 조인 결과가 특정 순서로 나올 것이라 기대하면 안 된다.
    - 정렬이 필요하면 항상 ORDER BY를 명시해야 한다.

### 11.4.8 GROUP BY

- 11.4.8.1 WITH ROLLUP
    - WITH ROLLUP은 GROUP BY 결과에 소계와 총계를 추가한다.
    - 엑셀 피벗 테이블의 소계, 총계와 비슷하다.
    - `GROUP BY dept_no WITH ROLLUP`은 부서별 건수 뒤에 전체 합계를 추가한다.
    - ROLLUP으로 생성된 소계 레코드의 그룹 칼럼 값은 기본적으로 NULL로 표시된다.
    - GROUP BY 칼럼이 2개 이상이면 소계 레벨도 계층적으로 늘어난다.
    - MySQL 8.0부터는 `GROUPING()` 함수로 ROLLUP이 만든 NULL을 사람이 읽기 좋은 문자열로 바꿀 수 있다.
- 11.4.8.2 레코드를 칼럼으로 변환해서 조회
    - GROUP BY와 집계 함수, `SUM(CASE WHEN ... THEN ... END)`를 이용하면 행을 열로 바꾸는 피벗 형태 결과를 만들 수 있다.
    - 부서별 사원 수를 `count_d001`, `count_d002`처럼 여러 칼럼으로 펼칠 수 있다.
    - 원리는 먼저 GROUP BY 결과를 만들고, 그 결과를 서브쿼리로 감싼 뒤 CASE WHEN과 SUM으로 한 행으로 모으는 것이다.
    - COUNT뿐 아니라 MIN, MAX, AVG, SUM에도 응용할 수 있다.
    - 단점은 열 이름이 쿼리에 고정되므로, 항목 추가나 변경이 생기면 SQL도 같이 바꿔야 한다.
    - 같은 방식으로 하나의 수치 칼럼을 조건별 여러 칼럼으로 분리할 수도 있다.
    - 예시에서는 사원의 입사 연도를 1980년대, 1990년대, 2000년대별로 나눠 부서별 집계 칼럼으로 만든다.
    - 간단한 SQL로 많은 프로그램 코드를 줄일 수 있고, WITH ROLLUP과 함께 쓰면 더 다양한 결과를 만들 수 있다.

### 11.4.9 ORDER BY

- 11.4.9.1 ORDER BY 사용법 및 주의사항
    - ORDER BY가 없으면 SELECT 결과 순서는 보장되지 않는다.
    - 인덱스를 사용한 조회면 인덱스 순서대로, 풀 스캔이면 저장 순서 비슷하게 나올 수 있지만 믿으면 안 된다.
    - 임시 테이블을 거치는 SELECT는 결과 순서를 더 예측하기 어렵다.
    - ORDER BY를 인덱스로 처리하지 못하면 별도 정렬 작업이 수행된다.
    - Extra의 `Using filesort`는 디스크 파일로 정렬했다는 뜻이 아니라 MySQL이 별도 정렬 알고리즘을 수행했다는 뜻으로 이해하는 편이 맞다.
    - 실제로 메모리만 썼는지, 디스크까지 썼는지는 상태 값으로 봐야 한다.
    - `SHOW STATUS LIKE 'Sort_%'`로 다음 값을 확인할 수 있다.
        - `Sort_merge_passes`
        - `Sort_range`
        - `Sort_scan`
        - `Sort_rows`
    - ORDER BY에는 칼럼명, 표현식, SELECT 목록 순번을 사용할 수 있다.
- 11.4.9.2 여러 방향으로 동시 정렬
    - MySQL 8.0부터는 여러 칼럼 정렬에서 ASC와 DESC를 섞은 인덱스를 만들 수 있다.
    - 예시처럼 `INDEX ix_salary_fromdate (salary DESC, from_date ASC)` 같은 정의가 가능하다.
    - 조회가 항상 내림차순이면 DESC 인덱스를 따로 만드는 편이 유리하다.
    - 정렬 방향 혼합은 8.0에서 유연해졌지만, 여전히 ORDER BY 방향과 인덱스 정의가 맞아야 인덱스를 탄다.
- 11.4.9.3 함수나 표현식을 이용한 정렬
    - `ORDER BY COS(salary)`처럼 연산 결과 기준 정렬도 가능하다.
    - MySQL 8.0 이전에는 가상 칼럼과 인덱스를 추가해야 했다.
    - MySQL 8.0부터는 함수 기반 인덱스를 지원해 표현식 정렬도 인덱스를 활용하도록 설계할 수 있다.

### 11.4.10 서브쿼리

- 서브쿼리는 SELECT 결과, FROM 절, WHERE 절 등 여러 위치에 사용할 수 있다.
- MySQL 5.6까지는 서브쿼리 최적화가 약한 경우가 많았지만, MySQL 8.0에서는 많이 개선되었다.
- 사용 위치에 따라 최적화 방식과 성능 영향이 다르다.
- 11.4.10.1 SELECT 절에 사용된 서브쿼리
    - SELECT 절 서브쿼리는 항상 칼럼 1개, 레코드 1개 결과만 반환해야 한다.
    - 결과가 0건이면 에러가 아니라 NULL이 반환된다.
    - 결과가 2건 이상이면 에러가 난다.
    - 칼럼이 2개 이상이어도 에러가 난다.
    - 즉 SELECT 절에는 스칼라 서브쿼리만 올 수 있다.
    - 조인으로 바꿔도 되는 쿼리를 SELECT 절 서브쿼리로 쓰는 경우가 많지만, 가능하면 조인으로 바꾸는 편이 더 빠르다.
    - 동일 레코드의 여러 칼럼을 얻기 위해 스칼라 서브쿼리를 반복하면 비효율적이다.
    - MySQL 8.0에서는 이런 패턴을 래터럴 조인 하나로 합칠 수 있다.
    - 다만 래터럴 조인도 내부 임시 결과와 추가 비용이 있어 실행 계획 비교가 필요하다.
- 11.4.10.2 FROM 절에 사용된 서브쿼리
    - 예전 MySQL에서는 FROM 절 서브쿼리 결과를 항상 임시 테이블로 저장한 뒤 다시 읽었다.
    - MySQL 5.7부터는 FROM 절 서브쿼리를 외부 쿼리와 병합하는 최적화가 가능하다.
    - EXPLAIN과 SHOW WARNINGS를 보면 실제로 병합되어 재작성된 쿼리를 확인할 수 있다.
    - 이 최적화는 뷰에도 적용된다.
    - 하지만 다음 기능이 서브쿼리에 있으면 외부 쿼리와 병합되지 못한다.
        - 집합 함수
        - DISTINCT
        - GROUP BY 또는 HAVING
        - LIMIT
        - UNION 또는 UNION ALL
        - SELECT 절 서브쿼리
        - 사용자 변수
    - FROM 절 서브쿼리에 ORDER BY가 있어도, 외부 쿼리에서 GROUP BY나 DISTINCT를 쓰면 그 정렬은 무시될 수 있다.
- 11.4.10.3 WHERE 절에 사용된 서브쿼리
    - WHERE 절 서브쿼리는 크게 세 가지로 구분한다.
        - 동등 또는 크다 작다 비교
        - IN 비교
        - NOT IN 비교
    - 11.4.10.3.1 동등 또는 크다 작다 비교
        - MySQL 5.5까지는 외부 쿼리를 먼저 읽으며 서브쿼리를 매번 평가하는 비효율적인 경우가 많았다.
        - 이후 버전부터는 보통 서브쿼리를 먼저 수행해 상수화하고, 그 결과로 외부 쿼리를 처리한다.
        - 단일 값 비교는 잘 최적화되어도, 튜플 비교는 여전히 주의가 필요하다.
        - 예시처럼 `(emp_no, from_date) = (subquery)` 형태는 외부 쿼리가 인덱스를 못 타고 풀 스캔될 수 있다.
    - 11.4.10.3.2 IN 비교 (`IN (subquery)`)
        - 실제 조인은 아니지만 세미 조인으로 처리된다.
        - MySQL 5.5까지는 최적화가 약해 풀 스캔이 많았다.
        - MySQL 5.6부터 8.0까지 세미 조인 최적화가 크게 발전했다.
        - MySQL은 조인 관계에 따라 다음 전략을 선택적으로 사용한다.
            - 테이블 풀-아웃
            - 퍼스트 매치
            - 루스 스캔
            - 구체화
            - 중복 제거
        - MySQL 8.0에서는 세미 조인 최적화에 너무 의존해 쿼리를 잘게 쪼개기보다, 엔진 기능을 적극 활용하는 방향이 더 낫다.
    - 11.4.10.3.3 NOT IN 비교 (`NOT IN (subquery)`)
        - 안티 세미 조인으로 처리된다.
        - 일반적인 Not-Equal 비교처럼 인덱스를 써도 성능이 잘 안 나오는 경우가 많다.
        - MySQL은 주로 다음 두 방식으로 최적화한다.
            - NOT EXISTS
            - 구체화
        - 두 방식 모두 큰 성능 향상을 보장하지는 않으므로, NOT IN 서브쿼리만 단독으로 둔 WHERE 조건은 풀 스캔을 피하기 어렵다.

### 11.4.11 CTE(Common Table Expression)

- CTE는 이름을 가진 임시 테이블처럼 SQL 문장 안에서 한 번 이상 참조할 수 있는 구문이다.
- CTE는 재귀적 반복 실행 여부를 기준으로 비 재귀적 CTE와 재귀적 CTE로 구분한다.
- MySQL에서 CTE는 단순 SELECT뿐 아니라 SELECT, UPDATE, DELETE, INSERT, REPLACE, CREATE TABLE, CREATE VIEW, DECLARE CURSOR, EXPLAIN 같은 문장의 앞부분에서도 사용할 수 있다.
- 11.4.11.1 비 재귀적 CTE(Non-Recursive CTE)
    - 문법은 `WITH cte1 AS (SELECT ...) SELECT ... FROM cte1` 형태다.
    - 같은 SELECT 안에서 여러 개의 CTE를 정의할 수 있다.
    - CTE는 FROM 절 서브쿼리와 비슷하게 보이지만, 동일한 임시 테이블을 여러 번 참조할 수 있다는 점이 다르다.
    - 같은 임시 결과를 여러 번 사용하는 경우 FROM 절 서브쿼리보다 CTE가 더 효율적일 수 있다.
    - 실행 계획 비교 예시에서는 CTE를 사용하면 동일한 derived 임시 테이블을 한 번만 만들지만, FROM 절 서브쿼리를 두 번 쓰면 각각 별도의 임시 테이블을 생성한다.
    - CTE를 재귀적으로 쓰지 않더라도 기존 FROM 절 서브쿼리보다 다음 장점이 있다.
        - CTE 임시 테이블은 재사용 가능하므로 FROM 절 서브쿼리보다 효율적이다.
        - CTE로 선언된 임시 테이블을 다른 CTE 쿼리에서 참조할 수 있다.
        - 같은 임시 테이블의 생성 부분과 사용 부분을 분리할 수 있어 가독성이 좋다.
- 11.4.11.2 재귀적 CTE(Recursive CTE)
    - 재귀적 CTE는 `WITH RECURSIVE cte AS (...)` 형태로 정의한다.
    - 기본 구조는 초기 결과를 만드는 비 재귀적 파트와, 직전 결과를 다시 참조해 다음 결과를 만드는 재귀적 파트로 구성된다.
    - 예시에서는 1부터 시작해 n+1을 반복 생성하면서 n < 5 조건까지 실행해 1부터 5까지의 결과를 만든다.

    - 재귀적 CTE는 다음 순서로 동작한다.
        - 비 재귀적 파트 실행
        - 결과를 CTE 임시 테이블에 저장
        - 직전 결과를 이용해 재귀적 파트 실행
        - 새 결과를 다시 CTE 임시 테이블에 저장
        - 더 이상 반환할 값이 없을 때까지 반복
    - 재귀적 CTE를 사용할 때 가장 흔한 실수는 반복 실행 종료 조건을 빠뜨리는 것이다.
    - 종료 조건을 빠뜨리면 무한 반복을 막기 위해 MySQL이 `cte_max_recursion_depth` 한도에서 중단한다.
    - 이 시스템 변수의 기본값은 1000이다.
    - 필요하면 SET cte_max_recursion_depth=...로 반복 횟수 상한을 조정할 수 있다.
- 11.4.11.3 재귀적 CTE(Recursive CTE) 활용
    - 재귀적 CTE는 단순 숫자 증가 예제뿐 아니라 계층형 데이터를 탐색하는 데 유용하다.
    - 첫 번째 활용 예시는 특정 사원에서 시작해 상위 조직장을 따라 올라가는 경로를 찾는 방식이다.

    - 이 예시에서는 비 재귀적 파트가 시작 사원을 고르고, 재귀적 파트가 manager_id를 따라 상위 직원과 계속 조인한다.
    - 두 번째 활용 예시는 최상위 조직장부터 시작해 하위 사원 조직도를 확장하는 방식이다.
        
        <img width="527" height="673" alt="image" src="https://github.com/user-attachments/assets/59199f4a-5c1d-4d9d-8f19-8a72399ce7cf" />

    - 이 예시에서는 manager_path 같은 경로 문자열을 누적해서 조직 구조를 한 줄 문자열로 표현한다.
    - 경로 문자열을 누적할 때는 재귀적 파트 결과가 더 길어지므로 칼럼 길이를 충분히 잡아야 한다.
    - 예시에서는 CAST(id AS CHAR(100)) 같은 방식으로 문자열 타입과 길이를 미리 넉넉히 맞춘다.
    - 그렇지 않으면 Data too long for column ... 오류가 발생할 수 있다.
    - 재귀적 쿼리는 다음과 같은 요구사항에 활용될 수 있다.
        - 단순 증가 수열을 가지는 임시 테이블 생성
        - 날짜 증가 형태의 임시 테이블 생성
        - 조직도 조회
        - BOM 탐색
        - 재귀적 쿼리 패턴이 필요한 기타 처리

### **11.4.12 윈도우 함수(Window Function)**

- 윈도우 함수는 조회된 결과 집합 안에서 행 단위 연산을 수행하면서도 각 원본 레코드를 그대로 유지할 수 있게 해주는 함수다.
- 집계 함수는 GROUP BY로 여러 레코드를 하나의 결과로 줄이지만, 윈도우 함수는 개별 레코드를 유지한 채 집계성 계산을 덧붙일 수 있다.
- 이 때문에 GROUP BY나 단순 집계로 해결하기 어려운 문제를 더 자연스럽게 풀 수 있다.
- 11.4.12.1 쿼리 각 절의 실행 순서
    
    <img width="409" height="261" alt="image" src="https://github.com/user-attachments/assets/8f814d21-05d7-44e1-bbaa-f6b086c8acc7" />

    - 윈도우 함수가 붙은 SELECT 결과는 FROM, WHERE, GROUP BY, HAVING이 모두 끝난 뒤 만들어진 중간 결과를 대상으로 계산된다.
    - 그 뒤 최종적으로 SELECT 절과 ORDER BY, LIMIT 절이 처리된다.
    - 그래서 GROUP BY 절이나 서브쿼리 내부의 LIMIT 결과를 기반으로 윈도우 함수를 적용하려면, 먼저 그 결과를 만드는 파생 테이블이나 서브쿼리가 필요하다.
    - 즉 윈도우 함수 계산 대상 범위는 쿼리에서 어디에 LIMIT이나 필터가 놓였는지에 따라 달라진다.
- 11.4.12.2 윈도우 함수 기본 사용법
    - 기본 문법은 `AGGREGATE_FUNC() OVER(<partition> <order>)` 형태다.
    - OVER() 절이 없으면 일반 집계 함수이고, OVER() 절이 붙으면 윈도우 함수가 된다.
    - PARTITION BY는 전체 결과를 여러 파티션으로 나누는 역할을 한다.
    - ORDER BY는 각 파티션 안에서 행의 순서를 정한다.
    - PARTITION BY나 ORDER BY가 필요 없는 경우에는 생략할 수 있다.
        
        <img width="353" height="284" alt="image" src="https://github.com/user-attachments/assets/f810c2c8-930b-4c5a-b432-7abf35bf487e" />

    - 파티션 안에서 연산할 범위를 더 세밀하게 지정하려면 frame을 사용한다.
    - frame 문법은 다음 형태다.
        - ROWS | RANGE ...
        - BETWEEN frame_start AND frame_end
    - frame 시작점과 끝점은 다음 값들을 사용할 수 있다.
        - CURRENT ROW
        - UNBOUNDED PRECEDING
        - UNBOUNDED FOLLOWING
        - expr PRECEDING
        - expr FOLLOWING
    - ROWS는 레코드 위치를 기준으로 범위를 정한다.
    - RANGE는 ORDER BY 값의 범위를 기준으로 범위를 정한다.
        
        <img width="687" height="433" alt="image" src="https://github.com/user-attachments/assets/1fd1186b-1b59-4204-b716-49af75d36231" />

    - 일부 윈도우 함수는 frame을 따로 명시하지 않으면 기본 frame이 적용된다.
    - [주의] ORDER BY 사용 시 기본 frame과 ORDER BY 미사용 시 기본 frame이 다르므로, 결과를 정확히 통제하려면 frame을 명시하는 편이 안전하다.
- 11.4.12.3 윈도우 함수
    - 윈도우 함수는 집계 함수 기반과 비 집계 함수 기반으로 나뉜다.
    - 집계 함수는 기존 집계 함수에 OVER 절을 붙여 사용할 수 있다.
    - 비 집계 함수는 순위, 이전 행, 다음 행, 누적 분포 같은 계산을 수행한다.
    - 11.4.12.3.1 DENSE_RANK()와 RANK(), ROW_NUMBER()
        - 세 함수 모두 ORDER BY 기준으로 순위를 계산한다.
        - RANK()는 동일 값이 있으면 같은 순위를 부여하고, 다음 순위는 건너뛴다.
        - DENSE_RANK()는 동일 값에 같은 순위를 주되, 다음 순위를 건너뛰지 않는다.
        - ROW_NUMBER()는 중복 여부와 관계없이 현재 정렬 순서대로 1부터 연속 번호를 부여한다.
    - 11.4.12.3.2 LAG()와 LEAD()
        - LAG()는 현재 행 기준으로 이전 행 값을 반환한다.
        - LEAD()는 현재 행 기준으로 다음 행 값을 반환한다.
        - 두 함수 모두 다음 인자를 받을 수 있다.
            - 반환할 칼럼
            - 몇 칸 앞뒤를 볼지 나타내는 offset
            - 값이 없을 때 사용할 기본값
    - 11.4.12.4 윈도우 함수와 성능
        - MySQL의 윈도우 함수는 아직 인덱스를 이용한 최적화가 부족한 부분이 있다.
        - 예시에서는 MAX(from_date) OVER(PARTITION BY emp_no)와 SELECT MAX(from_date) ... GROUP BY emp_no를 비교한다.
        - 실행 계획상 윈도우 함수 쿼리는 인덱스를 풀 스캔하고 Using filesort가 발생했다.
        - 반면 GROUP BY 쿼리는 프라이머리 키를 사용해 Using index for group-by로 처리된다.
        - 상태 값 비교에서도 윈도우 함수 쿼리가 훨씬 많은 레코드 읽기와 이동을 수행한 것으로 나타난다.
        - GROUP BY 쿼리는 유니크한 emp_no별 결과만 만들면 되지만, 윈도우 함수 쿼리는 모든 레코드에 대해 결과 칼럼을 붙여야 하므로 작업량이 훨씬 크다.
        - 다만 윈도우 함수는 SQL 표현력이 높고, GROUP BY로는 얻기 어려운 결과를 원본 행과 함께 반환할 수 있다는 장점이 있다.

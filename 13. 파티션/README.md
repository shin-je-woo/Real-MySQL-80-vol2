# 13. 파티션

## 13.1 개요

### 13.1.1 파티션을 사용하는 이유

- 파티션은 논리적으로 하나의 테이블을 물리적으로 여러 개의 작은 테이블처럼 나눠 관리하는 기능이다. 대용량 테이블에서 데이터와 인덱스를 더 작은 단위로 쪼개 관리할 수 있게 해준다. 다만 대용량 테이블에 적용한다고 항상 빨라지는 것은 아니며, 쿼리 특성에 따라 성능이 더 나빠질 수도 있다.
- 단일 INSERT와 단일 또는 범위 SELECT의 빠른 처리
    - 테이블이 너무 커져 인덱스의 워킹 셋이 메모리보다 커지면 SELECT뿐 아니라 INSERT, UPDATE, DELETE도 함께 느려질 수 있다.
    - 파티션은 데이터와 인덱스를 함께 조각내어 자주 접근하는 일부만 메모리에 올리기 쉽게 만든다.
        
        <img width="274" height="350" alt="image" src="https://github.com/user-attachments/assets/87e01693-d7f8-4474-a562-c11811a50e86" />

- 데이터의 물리적 저장소 분리
    - 데이터 파일이나 인덱스 파일의 크기가 너무 커지면 백업과 관리가 어려워진다.
    - 파티션을 사용하면 파일 크기를 조절하거나 저장 위치를 나눠 관리하는 데 도움이 된다.
    - 다만 MySQL은 파티션별로 서로 다른 인덱스를 두거나 파티션 단위로 전혀 다른 인덱스 구조를 두는 방식은 지원하지 않는다.
- 이력 데이터의 효율적인 관리
    - 로그처럼 일정 시간이 지나면 불필요해지는 데이터는 배치 DELETE보다 파티션 단위 삭제가 훨씬 단순하고 빠르다.
    - 파티션을 미리 기간 단위로 나눠두면 오래된 데이터는 파티션 DROP, 앞으로 저장할 데이터는 파티션 ADD로 처리할 수 있다.

### 13.1.2 MySQL 파티션의 내부 처리

- 파티션 테이블은 `PARTITION BY RANGE ( YEAR(reg_date) )`로 정의되며, `reg_date`에서 연도 값을 추출해 저장 파티션을 결정한다. 즉 파티션 키는 `reg_date` 자체가 아니라 `YEAR(reg_date)` 표현식이다.
- 레코드 INSERT
    - INSERT가 실행되면 MySQL 서버는 파티션 키 식을 평가해 레코드가 저장될 파티션을 결정한다.
    - 이 동작은 파티션되지 않은 일반 테이블과 달리, 저장 직전에 파티션 선택 단계가 한 번 더 수행된다는 점이 핵심이다.
        
        <img width="535" height="414" alt="image" src="https://github.com/user-attachments/assets/5a271f4f-4167-470e-ab16-ba34f0629051" />

- UPDATE
    - UPDATE는 변경 대상 레코드가 어느 파티션에 있는지 먼저 찾아야 한다.
    - 파티션 키 이외의 컬럼만 변경되면 기존 파티션 안에서 일반 UPDATE처럼 처리된다.
    - 파티션 키 컬럼이 변경되면 기존 파티션에서 레코드를 삭제한 뒤, 변경된 파티션 키 값을 다시 계산해 새 파티션에 INSERT하는 방식으로 처리된다.
        
        <img width="583" height="441" alt="image" src="https://github.com/user-attachments/assets/b86a9970-47af-40c2-99ad-4772526a496b" />

- 검색
    - 파티션 테이블 검색 성능은 다음 두 조건에 의해 좌우된다.
        - WHERE 조건으로 읽어야 할 파티션을 줄일 수 있는가
        - 각 파티션 내부에서 인덱스를 효율적(인덱스 레인지 스캔)으로 사용할 수 있는가
    - 파티션 선택 가능 + 인덱스 사용 가능: 가장 이상적이다. 필요한 파티션만 읽고, 그 안에서 인덱스 레인지 스캔을 수행한다.
    - 파티션 선택 불가 + 인덱스 사용 가능: 모든 파티션에 대해 인덱스 레인지 스캔을 수행하고 결과를 합친다.
    - 파티션 선택 가능 + 인덱스 사용 불가: 필요한 파티션만 읽지만 내부에서는 풀 스캔이 발생한다.
    - 파티션 선택 불가 + 인덱스 사용 불가: 모든 파티션을 풀 스캔하므로 가장 비효율적이다.
- 파티션 테이블의 인덱스 스캔과 정렬
    - 파티션 테이블의 인덱스는 모두 로컬 인덱스다.
    - 즉, 각 파티션별로 인덱스가 따로 생성되며, 테이블 전체를 아우르는 글로벌 인덱스는 지원하지 않는다.
    - 그래서 여러 파티션을 읽는 인덱스 스캔 결과는 전역적으로 바로 정렬되어 있지 않다. MySQL은 파티션별 결과를 우선순위 큐 방식으로 병합한다.
        
        <img width="429" height="384" alt="image" src="https://github.com/user-attachments/assets/dd8fc280-a5fe-4b09-b1a2-3c366e4116e9" />

- 파티션 프루닝
    - 옵티마이저가 쿼리에 필요 없는 파티션을 미리 제외하는 최적화를 파티션 프루닝이라고 한다.
    - EXPLAIN의 `partitions` 컬럼을 보면 실제 접근한 파티션만 표시된다.

## 13.2 주의사항

### 13.2.1 파티션의 제약 사항

- `PARTITION BY RANGE`는 레인지 파티션을 의미하고, 예시에서는 `YEAR(reg_date)`가 파티션 표현식이다. 즉 `reg_date`에서 연도만 추출해 연도별 범위로 나눈다.
- 파티션 기능에는 다음 제약이 있다.
    - 스토어드 루틴, UDF, 사용자 변수는 파티션 표현식에 사용할 수 없다.
    - 파티션 표현식은 컬럼 자체 또는 MySQL 내장 함수를 사용할 수 있지만, 일부 함수는 파티션 프루닝을 지원하지 않을 수 있다.
    - **프라이머리 키를 포함한 모든 유니크 인덱스는 파티션 키 컬럼을 포함해야 한다.**
    - 모든 인덱스는 로컬 인덱스다.
    - 같은 테이블의 모든 파티션은 동일한 스토리지 엔진만 사용할 수 있다.
    - 최대 8192개의 파티션을 가질 수 있다.
    - 파티션 생성 후 `sql_mode` 변경은 데이터 일관성을 깨뜨릴 수 있다.
    - **외래 키를 사용할 수 없다.**
    - 전문 검색 인덱스를 생성할 수 없다.
    - 공간 데이터 타입 컬럼은 파티션 테이블에 사용할 수 없다.
    - TEMPORARY 테이블은 파티션 기능을 사용할 수 없다.
- 파티션 표현식에 사용할 수 있는 연산자와 함수도 제한된다. 기본 산술 연산자와 일부 내장 함수는 허용되지만, 허용된다고 해서 모두 프루닝까지 잘 지원하는 것은 아니다. 설계 단계에서 실제 프루닝이 되는지 확인해야 한다.

### 13.2.2 파티션 사용 시 주의사항

- 파티션에서 가장 큰 제약은 **유니크 키**와의 관계다. 파티션의 목적은 작업 범위를 줄이는 것이지만, 유니크 인덱스는 중복 검사를 위해 전체 대상에 대한 일관된 판단이 필요하다. 그래서 모든 유니크 인덱스는 파티션 키를 포함해야 한다.
- 파티션과 유니크 키
    - 유니크 키가 파티션 키를 포함하지 않으면, 특정 데이터가 어느 파티션에 저장되어 있는지(혹은 저장될지) 특정할 수 없다.
    - 이 경우 중복 여부를 확인하기 위해 모든 파티션을 스캔해야 하므로 유니크 제약 조건을 보장할 수 없게 된다.
    - 이는 프라이머리 키(PK)에도 동일하게 적용되는 원칙이다.
    - 따라서 유니크 인덱스는 반드시 파티션 키를 포함하는 형태로 설계해야 한다.
    - 그래야 데이터의 중복 검증 범위가 단일 파티션 내로 국한될 수 있다.
- `open_files_limit` 설정
    - MySQL은 일반적으로 테이블을 파일 단위로 관리하므로 파티션 수가 많아질수록 동시에 열어야 하는 파일 수도 늘어난다.
    - 파티션되지 않은 일반 테이블보다 파티션 테이블은 열린 파일 수 요구량이 훨씬 커질 수 있어 `open_files_limit` 값을 충분히 크게 잡아야 한다.

## 13.3 MySQL 파티션의 종류

### 13.3.1 레인지 파티션

- 레인지 파티션은 파티션 키의 연속된 범위로 파티션을 정의하는 방식이다. 가장 일반적으로 사용하는 방법이며, `MAXVALUE`를 이용해 마지막 나머지 범위를 담는 파티션을 둘 수 있다.
- 용도
    - 날짜 기준으로 데이터가 누적되고, 연도, 월, 일 단위로 분석하거나 삭제해야 할 때
    - 범위 기준으로 데이터를 여러 파티션에 균등하게 나눌 수 있을 때
    - 파티션 키 위주의 검색이 자주 실행될 때
- 장점은 크게 두 가지다.
    - 큰 테이블을 작은 파티션으로 분리할 수 있다.
    - 필요한 파티션만 접근할 수 있다.
- 테이블 생성
    
    <img width="323" height="273" alt="image" src="https://github.com/user-attachments/assets/829185e1-1030-428c-87ea-4b45810aa61c" />

    - `PARTITION BY RANGE ( YEAR(hired) )`처럼 범위를 기준으로 파티션을 정의한다.
    - `VALUES LESS THAN`은 지정한 값보다 작은 값만 해당 파티션에 저장함을 의미한다.
    - `VALUES LESS THAN MAXVALUE`는 앞에서 다루지 못한 나머지 모든 값을 저장하는 마지막 파티션이다.
- 분리와 병합
    - 단순 파티션 추가`MAXVALUE` 파티션이 이미 있으면 단순 `ADD PARTITION`으로 중간 범위를 추가할 수 없다. 이 경우 `REORGANIZE PARTITION`을 사용해 기존 `MAXVALUE` 파티션을 둘로 나눠야 한다.
        
        <img width="380" height="129" alt="image" src="https://github.com/user-attachments/assets/8d22e35f-0996-40d9-9dd0-f76b4895d47c" />

- 파티션 삭제
    - 레인지 파티션에서는 `DROP PARTITION`으로 특정 파티션을 삭제할 수 있다. 보통 가장 오래된 구간을 통째로 제거하는 데 사용한다.
- 기존 파티션 분리
    - 하나의 파티션을 둘 이상으로 나누려면 `REORGANIZE PARTITION`을 사용한다. 기존 파티션의 모든 레코드를 새 분할 규칙에 따라 다시 배치해야 하므로 시간이 오래 걸릴 수 있다.
- 기존 파티션 병합
    - 여러 파티션을 하나로 합칠 때도 `REORGANIZE PARTITION`을 사용한다. 이 역시 데이터 재배치가 발생하므로 온라인 DDL이라도 읽기 잠금이 필요할 수 있다.

### 13.3.2 리스트 파티션

- 리스트 파티션은 파티션 키 값 하나하나를 직접 나열해 파티션을 구성하는 방식이다. 범위가 아니라 개별 값 집합으로 파티션을 나눈다는 점이 레인지 파티션과 다르다.
- 용도
    - 파티션 키 값이 코드값이나 카테고리처럼 고정적일 때
    - 키 값이 연속적이지 않아 정렬 순서와 무관하게 묶어야 할 때
    - 파티션 키 기준으로 데이터 건수를 고르게 나누거나, 특정 조건에서 파티션 키가 자주 사용될 때
- 테이블 생성`PARTITION BY LIST(category_id)`처럼 정의하고, 각 파티션마다 `VALUES IN (...)` 형태로 값을 직접 나열한다. 마지막 파티션에 `NULL`을 포함시키는 것도 가능하다. 레인지 파티션과 달리 `MAXVALUE` 파티션은 정의할 수 없다.
- 리스트 파티션은 정수 타입뿐 아니라 문자열 타입 컬럼도 파티션 키로 사용할 수 있다.
- 분리와 병합
    - 리스트 파티션은 `VALUES LESS THAN`이 아니라 `VALUES IN`을 사용한다는 점만 다르고, 파티션을 나누거나 합칠 때는 레인지 파티션과 마찬가지로 `REORGANIZE PARTITION`을 사용한다.
- 주의사항
    - 명시되지 않은 나머지 값을 저장하는 `MAXVALUE` 파티션은 만들 수 없다.
    - 레인지 파티션과 달리 `NULL`을 저장하는 별도 파티션을 만들 수 있다.

### 13.3.3 해시 파티션

- 해시 파티션은 MySQL이 정의한 해시 계산 결과에 따라 레코드가 저장될 파티션을 정하는 방식이다. 파티션 키 값은 정수형 표현식을 사용해야 한다.
- 용도
    - 레인지나 리스트처럼 의미 있는 구간으로 나누기 어려울 때
    - 전체 레코드가 비슷한 사용 빈도를 보이고, 특정 범위보다 균등 분산이 더 중요할 때
- 테이블 생성`PARTITION BY HASH(id) PARTITIONS 4`처럼 정의한다. 파티션 개수는 `PARTITIONS n`으로 지정한다. 파티션 이름을 직접 지정할 수도 있지만, 개수만 지정하면 기본적으로 `p0`, `p1`, `p2` 형식으로 자동 생성된다.
- 해시 파티션의 분리와 병합은 레인지, 리스트 파티션과 다르다. 특정 파티션만 분리하거나 병합하는 것이 아니라, 파티션 수가 바뀌면 전체 파티션에 저장된 레코드를 다시 재배치해야 한다.
- 파티션 추가
    - 새 파티션을 추가하면 기존 모든 파티션의 레코드가 새 파티션 개수 기준으로 다시 분산된다. INPLACE 알고리즘이라도 실제로는 테이블 리빌드와 레코드 재배치가 발생하므로 부하가 크고, 다른 트랜잭션의 DML도 허용되지 않는다.
        
        <img width="349" height="210" alt="image" src="https://github.com/user-attachments/assets/8cbfc196-0735-4689-8e8f-7264fae80aca" />

- 파티션 삭제
    - 해시 파티션에서는 특정 파티션만 `DROP PARTITION`으로 삭제할 수 없다. RANGE/LIST에서만 허용된다.
- 파티션 분할
    - 특정 파티션 하나를 둘 이상으로 나누는 기능은 없다. 전체 파티션 개수를 늘리는 방식만 가능하다.
- 파티션 병합
    - 두 개 이상의 파티션을 하나로 합치는 기능은 없고, `COALESCE PARTITION n`으로 전체 파티션 개수를 줄인다. 이때도 테이블 전체 레코드 재배치가 필요하다.
        
        <img width="359" height="222" alt="image" src="https://github.com/user-attachments/assets/6df70bb9-0c05-4577-9d1b-68f8909e3ed3" />

- 주의사항
    - 특정 파티션만 삭제하는 것은 불가능하다.
    - 새 파티션 추가는 단순 추가가 아니라 전체 데이터 재배치다.
    - 레인지나 리스트 파티션과 관리 방식이 많이 다르므로, 용도에 맞는지 먼저 확인해야 한다.
    - 사용자가 기대하는 일부 파티션 조작 특성은 대부분 레인지나 리스트에 해당하며, 해시는 다르게 동작한다.

### 13.3.4 키 파티션

- 키 파티션은 해시 파티션과 비슷하지만, 사용자가 직접 해시 식을 작성하지 않고 MySQL 서버가 내부적으로 값을 해싱해 파티션을 결정한다. MySQL은 선택한 파티션 키 값을 MD5 기반으로 처리한 뒤 MOD 연산으로 분산한다.
- 테이블 생성`PARTITION BY KEY(...) PARTITIONS n` 형태로 정의한다.
    - 프라이머리 키가 있으면 괄호를 비워 두었을 때 프라이머리 키 전체가 파티션 키가 된다.
    - 프라이머리 키가 없고 유니크 키가 있으면 유니크 키 전체가 사용된다.
    - 프라이머리 키나 유니크 키를 구성하는 컬럼 중 일부만 직접 지정할 수도 있다.
- 주의사항 및 특이사항
    - 키 파티션은 내부적으로 MD5를 사용하므로 파티션 키가 반드시 정수형일 필요는 없다.
    - 프라이머리 키나 유니크 키를 구성하는 컬럼 중 일부만 파티션 키로 사용할 수 있다.
    - 유니크 키를 파티션 키에 사용했다면 해당 유니크 키는 NOT NULL이어야 한다.
    - 해시 파티션보다 파티션 간 레코드 분산이 더 고르게 되는 편이다.

### 13.3.5 리니어 해시 파티션/리니어 키 파티션

- 해시 파티션이나 키 파티션은 파티션 수가 바뀔 때 전체 재배치가 발생하는 단점이 있다. 이를 줄이기 위해 리니어 방식은 `Power-of-two` 알고리즘을 사용해 파티션 추가와 통합 시 영향 범위를 줄인다.
- 추가 및 통합
    - 리니어 해시/키 파티션에서 새 파티션을 추가하면 전체가 아니라 일부 특정 파티션의 레코드만 재분배된다.
        
        <img width="360" height="211" alt="image" src="https://github.com/user-attachments/assets/a74d7c29-31d3-4ad0-a364-5dd2a00180e6" />

    - 파티션을 통합할 때도 일부 파티션만 이동 대상이 되고, 나머지는 그대로 유지된다.
        
        <img width="392" height="213" alt="image" src="https://github.com/user-attachments/assets/366bc806-ae4b-4ac1-9612-05a50463ee2a" />

- 관련 주의사항
    - 일반 해시/키 파티션보다 파티션 추가와 통합 시 영향 범위를 줄일 수 있다.
    - 하지만 레코드 분산 균등성은 일반 해시/키보다 떨어질 수 있다.
    - 파티션을 자주 늘리거나 줄여야 하면 리니어 방식이 유리하고, 파티션 조정이 거의 없다면 일반 방식이 더 적합할 수 있다.

### 13.3.6 파티션 테이블의 쿼리 성능

- 파티션 테이블 성능의 핵심은 파티션 프루닝이다. 모든 파티션을 읽는지, 필요한 일부 파티션만 읽는지가 성능에 큰 영향을 준다. 실행 계획에서 어떤 파티션이 접근 대상인지 확인해야 한다.
- 레인지나 리스트 파티션은 보통 파티션을 명시적으로 설계하므로 수십 개 수준에서 운영하는 경우가 많다. 반면 해시나 키 파티션은 매우 많은 수의 파티션을 쉽게 만들 수 있다.
- `PARTITION BY KEY() PARTITIONS 1024`로 구성된 테이블에서 `name='toto'`를 찾는다면, 파티션되지 않은 일반 테이블에서는 B-Tree를 한 번 탐색하면 되지만, 파티션된 테이블에서는 내부적으로 1024개 파티션에 나뉜 인덱스를 각각 고려해야 한다. 그래서 작은 테이블을 균등한 사용 패턴으로 지나치게 많이 파티션하면 오히려 오버헤드가 커진다.
- 테이블을 10개 파티션으로 나눴을 때 주로 1개나 2개만 읽는다면 도움이 될 수 있다. 하지만 10개 모두를 비슷하게 읽는다면 성능 개선보다 오버헤드가 더 클 수 있다. MySQL 파티션은 샤딩이 아니므로, 단순히 여러 서버에 나누어 저장하는 효과를 기대하면 안 된다.

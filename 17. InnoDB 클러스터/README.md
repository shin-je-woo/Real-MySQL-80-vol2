# 17. InnoDB 클러스터

## 17.1 InnoDB 클러스터 아키텍처

### 17.1.1 InnoDB 클러스터의 의미와 구성 요소

- InnoDB 클러스터는 단순한 MySQL 서버 기능 하나가 아니라 MySQL 고가용성을 위해 여러 구성 요소를 묶은 집합이다.
    
    <img width="305" height="477" alt="image" src="https://github.com/user-attachments/assets/13a793c9-50e6-439a-a331-fecfb9915c2c" />

- 핵심 구성 요소는 다음 3가지다.
    - 그룹 복제(Group Replication)
    - MySQL 라우터(MySQL Router)
    - MySQL 셸(MySQL Shell)
- 그룹 복제는 데이터 복제뿐 아니라 멤버 추가와 제거 같은 그룹 멤버십 관리까지 담당한다.
- MySQL 라우터는 애플리케이션과 MySQL 서버 사이에서 프록시처럼 동작하며 적절한 서버로 쿼리를 전달한다.
- MySQL 셸은 SQL 실행만 하는 클라이언트가 아니라 자바스크립트, 파이썬 기반 관리 기능과 AdminAPI를 제공해 클러스터 생성과 운영을 담당한다.

### 17.1.2 기존 복제 기반 HA와의 차이

- 단순 소스-레플리카 복제만으로는 자동 페일오버가 되지 않는다.
- 장애 시에는 새 소스 승격, 기존 소스 쓰기 차단, 애플리케이션 커넥션 변경 같은 작업이 필요하며 수동 처리 시 시간이 든다.
- InnoDB 클러스터는 이런 작업을 자동화하는 빌트인 HA 솔루션이라는 점이 핵심이다.
- 별도 서드파티 HA 솔루션과 달리 MySQL이 공식 제공하고 함께 테스트되는 패키지라는 점이 장점이다.

### 17.1.3 InnoDB 클러스터 내부 서버 역할

- 클러스터에 포함된 MySQL 서버는 그룹 복제 기반으로 묶이며 읽기와 쓰기가 가능한 프라이머리 또는 읽기만 가능한 세컨더리 역할을 가진다.
- 그룹 복제 모드에 따라 프라이머리가 1대일 수도 있고 여러 대일 수도 있다.
- InnoDB 스토리지 엔진만 사용 가능하다.
- 고가용성을 위해 최소 3대 이상 구성이 권장된다. 3대면 1대 장애가 나도 과반을 유지할 수 있기 때문이다.

### 17.1.4 Router와 Shell의 역할

- 클라이언트는 보통 MySQL 서버에 직접 붙지 않고 MySQL 라우터를 통해 접속한다.
- 라우터는 클러스터 메타데이터를 기반으로 적절한 MySQL 서버로 쿼리를 전달하므로 클라이언트는 서버 구성 변경을 몰라도 된다.
- MySQL 셸은 클러스터 생성, 관리, 설정 변경을 담당한다. 단순 쿼리는 라우터를 통해 처리할 수도 있다.
- 장애가 발생하면 그룹 복제가 멤버를 제외하고, 라우터가 이를 인식해 클라이언트 요청을 정상 멤버에게만 보낸다.
- 결과적으로 기존 복제 환경에서 수동으로 하던 장애 조치와 커넥션 전환이 자동화된다.

## 17.2 그룹 복제(Group Replication)

### 17.2.1 그룹 복제의 개념과 특징

- 그룹 복제는 여러 MySQL 서버가 하나의 복제 그룹을 이루어 서로 통신하며 복제를 처리한다.
- 기존 복제가 소스와 레플리카의 단방향 구조라면 그룹 복제는 그룹 멤버 간 통신과 합의를 기반으로 동작한다.
    
    <img width="471" height="266" alt="image" src="https://github.com/user-attachments/assets/f4f5b4af-72d3-418b-b610-df3d1dab2128" />

- 그룹 복제는 기존 비동기, 반동기 복제와 달리 합의와 인증 절차가 트랜잭션 처리 경로에 포함된다.
    
    <img width="521" height="581" alt="image" src="https://github.com/user-attachments/assets/607ff827-0498-4a47-8b43-919cfc3c81f5" />

- 그룹 복제가 제공하는 대표 기능은 다음과 같다.
    - 그룹 멤버 관리
    - 그룹 단위 정렬된 트랜잭션 적용
    - 트랜잭션 충돌 감지
    - 자동 페일오버
    - 자동 분산 복구

### 17.2.1.1 그룹 복제 아키텍처

- 그룹 복제는 별도 플러그인으로 구현된다.
- 사용 시 `group_replication_applier` 채널이 생성되며 그룹에서 실행된 트랜잭션을 전달받아 적용한다.
- 새 멤버가 가입해 기존 데이터를 가져와야 할 때는 `group_replication_recovery` 채널이 생성되어 분산 복구를 수행한다.
    
    <img width="294" height="278" alt="image" src="https://github.com/user-attachments/assets/2e83b53c-f274-466a-b075-234287b05954" />

    <img width="400" height="296" alt="image" src="https://github.com/user-attachments/assets/3b9ce78e-78c7-428d-a13e-d7e064362b22" />

- 내부는 대략 다음 계층으로 나뉜다.
    - MySQL 서버와 상호작용하는 Plugin API 계층
    - 실제 그룹 복제 기능을 수행하는 Replication Plugin 계층
    - 그룹 통신 시스템 API
    - 그룹 통신 엔진 XCom
- XCom은 그룹 멤버 간 통신, 동일 순서 전달 보장, 장애 감지, 합의 처리를 담당한다.
- 합의 처리에는 Paxos 계열 알고리즘이 사용된다.
- 그룹 복제는 과반수 합의를 전제로 하므로 허용하려는 장애 서버 수를 f라고 할 때 전체 서버 수 n은 `n = 2f + 1`을 만족하도록 구성해야 한다.

### 17.2.2 그룹 복제 모드

### 17.2.2.1 싱글 프라이머리 모드

- 쓰기를 처리하는 프라이머리 서버가 1대만 존재하는 모드다.
- 클러스터를 처음 싱글 프라이머리 모드로 구성하면 구성 작업을 진행한 서버가 프라이머리로 지정된다.
- 프라이머리가 아닌 서버는 `super_read_only=ON` 상태로 읽기 전용이 된다.
    
    <img width="358" height="311" alt="image" src="https://github.com/user-attachments/assets/fa01065a-1997-41a7-a141-9c3e59287aa2" />

- 프라이머리 변경은 다음 상황에서 일어난다.
    - 현재 프라이머리가 자발적 또는 비자발적으로 그룹을 탈퇴한 경우
    - `group_replication_set_as_primary()` UDF로 특정 멤버를 지정한 경우

### 17.2.2.2 멀티 프라이머리 모드

- 그룹 멤버 전부가 프라이머리로 동작해 어느 서버로도 읽기와 쓰기 요청을 보낼 수 있는 모드다.
    
    <img width="379" height="350" alt="image" src="https://github.com/user-attachments/assets/f99f16f6-628f-4d68-99bc-0abbccc047dd" />

- 모든 멤버에서 쓰기가 발생할 수 있으므로 멤버 간 MySQL 버전 호환성이 매우 중요하다.
- 새 멤버 참여 시 버전 호환성을 검사하며, 기준에 따라 참여 가능 여부와 읽기 전용 유지 여부가 결정된다.
- 정리하면 다음과 같다.
    - 그룹 내 최저 버전보다 낮은 버전 멤버는 참여할 수 없다.
    - 그룹 내 최저 버전과 같은 버전이면 정상 참여 가능하다.
    - 멀티 프라이머리 그룹에서 더 높은 버전 멤버는 참여 가능하지만 읽기 전용 모드로 유지될 수 있다.
- 멤버가 탈퇴하면 가장 낮은 버전을 다시 확인해, 필요하면 자동으로 싱글 프라이머리 전환이나 읽기 전용 해제를 수행한다.
- `group_replication_switch_to_multi_primary_mode()` 같은 UDF로 모드 전환이 가능하다.

# 12. 확장 검색

## 12.1 전문 검색

- 일반적인 B-Tree 인덱스는 짧은 키 비교에는 적합하지만, 긴 문서를 단어 단위로 잘라 검색하는 용도에는 한계가 있다.
- 이를 보완하는 기능이 전문 검색(Full-text Search)이다.
- MySQL 8.0에서는 InnoDB에서도 전문 검색을 사용할 수 있다.
- 한글처럼 형태소 분석이 어려운 언어에서는 n-gram 방식이 특히 중요하다.

### 12.1.1 전문 검색 인덱스의 생성과 검색

- MySQL은 전문 검색 토큰 분리에 2가지 계열을 사용한다.
    - 형태소 분석 계열
        - 공백 같은 구분자를 기준으로 단어를 나누는 방식에 가깝다.
        - 실제 형태소 분석기나 어근 분석기를 MySQL 서버가 완전하게 제공하는 것은 아니다.
    - n-gram 파서
        - 문자열을 고정 길이 n의 토큰으로 잘라 인덱싱한다.
        - 한글 검색에 더 실용적이다.
- n-gram의 핵심 설정은 `ngram_token_size`다.
    - `2`면 bi-gram이다.
    - `3`이면 tri-gram이다.
    - 보통 bi-gram 또는 tri-gram이 주로 사용된다.
    - 서버 시작 시 설정을 읽기 때문에 변경 시 재시작이 필요하다.
    - 인덱스 생성 시점과 검색 시점의 값이 일치해야 결과가 어긋나지 않는다.
- n-gram 전문 검색 인덱스를 만들 때는 `WITH PARSER ngram`을 명시해야 한다.
    - 이를 생략하면 기본 파서가 사용된다.
    - 즉, n-gram 검색을 기대하고 인덱스를 만들었더라도 옵션이 없으면 전혀 다른 방식으로 토큰이 저장된다.
- bi-gram, tri-gram 예제로 드러나는 규칙은 다음과 같다.
    - 검색어 길이가 `ngram_token_size`보다 짧으면 검색할 수 없다.
    - 검색어 길이가 `ngram_token_size`와 같거나 더 길면 검색할 수 있다.
    - 검색은 단어의 시작 부분뿐 아니라 중간과 끝 부분도 가능하다.
    - 예를 들어 토큰 크기 2라면 2글자 이상 검색이 가능하고, 토큰 크기 3이라면 3글자 이상 검색이 가능하다.
- n-gram 검색은 문서 저장 시에도 토큰 분해가 일어나고, 검색 시에도 검색어를 같은 길이로 다시 분해해 비교한다.
    - 이 때문에 `ngram_token_size`는 저장과 조회 모두에 영향을 준다.
    - 이미 인덱스를 만든 뒤 값을 바꾸면 검색 결과가 기대와 달라질 수 있다.
- n-gram 토큰 예제에서는 단어가 다음처럼 잘린다.

    <img width="570" height="262" alt="image" src="https://github.com/user-attachments/assets/d2d27b02-14bd-4492-85bf-3edfe23ac741" />

    - "단편적인"은 여러 조각으로 분해된다.
    - "MySQL" 같은 영문 단어도 고정 길이 토큰으로 잘린다.
    - `ngram_token_size`보다 짧은 단어는 토큰으로 저장되지 않는다.
- 검색 처리 흐름은 다음과 같다.
    - 검색어를 n-gram 길이에 맞춰 분해한다.
    - 분해된 각 토큰을 전문 검색 인덱스로 찾는다.
    - 결과를 문서 ID 단위로 묶는다.
    - 각 문서 안에서 토큰의 위치와 포함 여부를 기준으로 최종 일치 여부를 계산한다.
- 실무상 기억할 점은 다음과 같다.
    - 한글에서는 `ngram_token_size=2`가 범용적으로 더 적합한 선택이 되기 쉽다.
    - 하지만 실제 검색어 길이 분포에 맞춰 값을 결정해야 한다.
    - 지나치게 긴 토큰 크기는 짧은 검색어를 거의 못 찾게 만든다.

### 12.1.2 전문 검색 쿼리 모드

- 전문 검색은 크게 3가지 모드를 지원한다.
    - 자연어 검색(NATURAL LANGUAGE MODE)
    - 불리언 검색(BOOLEAN MODE)
    - 검색어 확장(WITH QUERY EXPANSION)

### 12.1.2.1 자연어 검색(NATURAL LANGUAGE MODE)

- 모드를 명시하지 않으면 기본적으로 자연어 검색이 사용된다.
- 검색어와 관련된 단어를 많이 포함한 순서대로 결과를 정렬해 반환한다.
- 결과에는 `score`가 함께 계산될 수 있으며, 관련도가 높은 문서가 앞에 온다.
- 문장을 그대로 검색어로 넣는 방식도 가능하다.
    - 이 경우 문장을 공백이나 구두점 기준으로 나누고, 다시 n-gram 토큰으로 분해해 관련도를 계산한다.
    - 완전히 같은 문장을 찾는 방식이라기보다, 분해된 토큰의 일치도를 바탕으로 점수를 계산한다.

### 12.1.2.2 불리언 검색(BOOLEAN MODE)

- 불리언 검색은 검색어 존재 여부를 논리 조건으로 제어할 수 있다.
- 주요 연산자는 다음과 같다.
    - `+` : 포함되어야 한다.
    - `-` : 포함되면 안 된다.
- 예제의 의미는 다음과 같다.
    - `+MySQL -manual`

        - `MySQL`은 있어야 하고 `manual`은 없어야 한다.
    - `+MySQL +manual`

        - 두 단어가 모두 있어야 한다.
- 큰따옴표로 묶은 구문은 하나의 구처럼 취급된다.

    - `"MySQL man"`은 `MySQL` 뒤에 `man`이 따라오는 형태를 찾는다.
    - `"MySQL doc"`은 토큰 순서가 맞지 않으면 결과가 없다.
- `+`, `-`를 전혀 쓰지 않으면 포함된 단어 중 하나만 있어도 일치로 판단한다.
    - 따라서 불리언 모드라고 해도 연산자를 생략하면 자연어 검색과 비슷하게 느껴질 수 있다.
    - 다만 점수 계산 방식은 동일하지 않다.
- 와일드카드 `*`도 사용할 수 있다.
    - 하지만 n-gram 전문 검색에서는 원래 단어 일부 검색이 가능하므로 효과가 거의 없다.
    - 즉, n-gram 인덱스를 쓰는 경우에는 굳이 와일드카드 검색을 사용할 필요가 없다.

### 12.1.2.3 검색어 확장(QUERY EXPANSION)

- 검색어 확장은 사용자가 입력한 검색어로 먼저 검색한 뒤, 그 결과에서 자주 함께 등장한 단어를 모아 다시 검색하는 방식이다.
- 목적은 연관성이 높은 결과를 더 넓게 찾는 것이다.
- 동작 흐름은 다음과 같다.
    - 사용자가 입력한 검색어로 1차 검색을 수행한다.
    - 1차 결과에서 연관 단어를 추출한다.
    - 추출된 단어를 포함해 다시 전문 검색을 수행한다.
    - 두 결과를 합쳐 최종 결과를 반환한다.
- 장점은 다음과 같다.
    - 사용자가 직접 연관어를 알지 못해도 넓은 결과를 얻을 수 있다.
- 단점은 다음과 같다.
    - 원래 검색어와 직접 관련 없는 결과가 섞일 수 있다.
    - 원하지 않는 문서가 많아질 수 있다.
    - 추가 검색 비용이 든다.
- 따라서 검색어 확장은 편리하지만, 정확도가 중요한 검색에서는 신중히 써야 한다.

### 12.1.3 전문 검색 인덱스 디버깅

- 전문 검색은 원문을 그대로 저장하는 일반 인덱스와 달리 토큰을 가공해 저장하므로, 왜 검색이 안 되는지 파악하기 어렵다.
- 이를 위해 MySQL은 InnoDB 전문 검색용 진단 테이블을 제공한다.
- 먼저 `innodb_ft_aux_table`에 대상 테이블을 지정해야 한다.
    - 지정한 뒤 `information_schema`의 여러 보조 테이블을 조회할 수 있다.
- 확인할 수 있는 주요 테이블은 다음과 같다.
    - `information_schema.innodb_ft_config`
        - stopword 사용 여부, 동기화 상태 등 전문 검색 설정 정보를 보여준다.
    - `information_schema.innodb_ft_index_table`
        - 어떤 토큰이 어떤 문서에 어떤 위치로 저장되었는지 보여준다.
    - `information_schema.innodb_ft_index_cache`
        - 아직 디스크에 반영되지 않은 메모리 캐시 상태를 보여준다.
    - `information_schema.innodb_ft_deleted`, `innodb_ft_being_deleted`
        - 삭제 처리된 문서 상태를 확인할 수 있다.
- 실무에서는 다음 순서로 보는 것이 좋다.
    - stopword나 토큰 크기 설정이 기대와 맞는지 확인한다.
    - 실제 토큰이 어떻게 저장됐는지 확인한다.
    - 새로 INSERT한 데이터가 캐시에만 있는지 확인한다.
    - 삭제된 문서가 검색 인덱스에서 어떻게 반영되는지 확인한다.

## 12.2 공간 검색

- MySQL 8.0에서는 공간 데이터 관리 기능이 많이 강화됐다.
- 공간 검색은 단순히 점 하나를 저장하는 문제가 아니라 좌표계, 단위, 거리 계산 방식, 인덱스 동작 방식을 함께 이해해야 한다.

### 12.2.1 용어 설명

- OGC(Open Geospatial Consortium)
    - 위치 기반 데이터 표준을 만드는 단체다.
- OpenGIS
    - GIS 표준을 의미하며 WKT, WKB, SRID 같은 표현 방식을 포함한다.
- SRS
    - 공간 참조 시스템이다.
    - 좌표를 어떤 체계로 해석할지 정의한다.
- GCS와 PCS
    - GCS는 지리 좌표계다.
        - 위도, 경도처럼 각도 단위를 사용한다.
    - PCS는 투영 좌표계다.
        - 평면에 투영된 좌표를 사용하며 보통 meter 단위를 사용한다.
    - 같은 위치라도 어떤 SRS를 쓰는지에 따라 값의 의미가 달라진다.
- SRID
    - 특정 SRS를 식별하는 번호다.
    - MySQL에서는 `SRS_ID` 컬럼이나 함수 인자로 등장한다.
- WKT와 WKB
    - WKT는 사람이 읽기 쉬운 텍스트 표현이다.
    - WKB는 저장과 전송에 적합한 바이너리 표현이다.
    - MySQL 내부 저장 포맷은 WKB와 유사하지만 완전히 동일하지는 않다.
- MBR과 R-Tree
    - MBR은 도형을 감싸는 최소 사각형이다.
    - 공간 인덱스는 이 MBR 기반 포함 관계를 빠르게 검사한다.
    - 이를 위해 R-Tree 구조가 사용된다.

### 12.2.2 SRS(Spatial Reference System)

- MySQL은 `information_schema.ST_SPATIAL_REFERENCE_SYSTEMS`를 통해 SRS 정보를 제공한다.
- 중요한 컬럼은 다음과 같다.
    - `SRS_ID`
        - 좌표계 식별자다.
    - `DEFINITION`
        - 좌표계가 어떤 체계인지 정의한 내용이다.
- 대표 예제는 다음과 같다.
    - `SRID=4326`
        - `WGS 84`
        - 지리 좌표계다.
        - 값의 단위는 각도다.
    - `SRID=3857`
        - `WGS 84 / Pseudo-Mercator`
        - 투영 좌표계다.
        - 값의 단위는 meter 기반이다.
- `DEFINITION`에서 `GEOGCS`는 지리 좌표계, `PROJCS`는 투영 좌표계를 의미한다.
- 또한 `AXIS` 순서도 중요하다.
    - WGS 84 예제에서는 위도, 경도 순으로 해석해야 하는 점이 강조된다.
    - 그래서 `ST_X()`, `ST_Y()`만으로 처리하면 혼동될 수 있다.
    - 지리 좌표계에서는 `ST_Latitude()`, `ST_Longitude()`를 쓰는 편이 더 안전하다.
- `SRID=0`은 특별한 기본 평면 좌표계처럼 취급된다.
    - 별도로 SRID를 지정하지 않은 공간 데이터는 MySQL 8.0에서 기본적으로 SRID 0으로 인식된다.
    - 이 경우 지리 좌표계 전용 함수는 사용할 수 없다.

### 12.2.3 투영 좌표계와 평면 좌표계

- MySQL은 지리 좌표계나 투영 좌표계 외에 평면 좌표계처럼 다루는 `SRID=0` 좌표도 사용할 수 있다.
- 평면 좌표계는 단위가 정해져 있지 않은 무한 평면 개념에 가깝다.
- 투영 좌표계는 지구를 평면으로 투영한 좌표계이며, 대표적으로 `SRID=3857`이 나온다.
- 평면 좌표계와 투영 좌표계의 차이는 다음과 같다.
    - SRID 0은 단위가 없는 평면 좌표다.
    - SRID 3857은 meter 단위를 가진 투영 좌표다.
    - 둘 다 평면 거리처럼 계산되지만, SRID 3857은 실제 지구 곡면을 완전하게 반영한 값은 아니다.
- 테이블 생성 시 컬럼에 SRID를 명시할 수 있다.
    - 예제에서는 `POINT SRID 0`, `POINT SRID 3857`처럼 사용한다.
    - 컬럼에 지정된 SRID와 다른 SRID 데이터를 INSERT하면 에러가 발생한다.
- 조회 시에는 다음 함수가 중요하다.
    - `ST_AsText()`
        - WKT로 변환한다.
    - `ST_AsWKB()`
        - WKB로 확인한다.
    - `ST_X()`, `ST_Y()`
        - 좌표값을 꺼낸다.
    - `ST_SRID()`
        - 객체의 SRID를 확인한다.
- 거리 계산 예제에서는 다음이 드러난다.
    - SRID 0이나 SRID 3857 모두 피타고라스 정리에 가까운 평면 거리 계산처럼 보인다.
    - SRID 0은 단위가 없고, SRID 3857은 meter 단위다.
    - 따라서 같은 수치처럼 보여도 해석은 다르다.

### 12.2.4 지리 좌표계

- 지리 좌표계는 GPS처럼 실제 지구상의 위치를 저장할 때 사용한다.
- 대표적으로 `SRID=4326`이 사용된다.
- 12.2.4.1 지리 좌표계 데이터 관리
    - 예제에서는 `sphere_coord` 테이블에 `POINT NOT NULL SRID 4326` 컬럼을 정의하고 공간 인덱스를 만든다.
    - 공간 인덱스를 만들려면 인덱스 대상 컬럼은 `NOT NULL`이어야 한다.
    - 거리 계산에는 `ST_Distance_Sphere()`를 사용한다.
        - 결과는 meter 단위다.
        - 지구 구면을 기준으로 거리를 계산한다.
        - 실제 반경 검색에 적합하다.
    - 특정 위치에서 반경 1km 이내를 찾는 기본 쿼리는 `ST_Distance_Sphere()`만으로도 작성할 수 있다.
        - 하지만 이 방식은 공간 인덱스를 활용하지 못하고 풀 테이블 스캔이 된다.
    - 그래서 먼저 MBR을 이용한 1차 필터링이 필요하다.
        - 기준점에서 반경 1km에 해당하는 상하좌우 범위를 계산한다.
        - 그 범위를 감싸는 사각형 `POLYGON`을 만든다.
        - 이 MBR 사각형으로 공간 인덱스를 타게 한다.
        - 이후 `ST_Distance_Sphere()`로 실제 반경 조건을 다시 적용한다.
- 12.2.4.2 지리 좌표계 주의 사항
- 12.2.4.2.1 정확성 주의 사항
    - `SRID=4326`에서 `ST_Contains()`가 기대와 다르게 동작하는 사례가 소개된다.
    - 회피 방법은 다음과 같다.
        - 단순 포함 관계만 믿지 않는다.
        - 다각형 중심점 기준 거리 조건을 추가한다.
        - `ST_Contains()`와 `ST_Distance_Sphere()`를 함께 사용해 정확도를 보정한다.
    - 공간 인덱스를 사용한 경우와 사용하지 않은 경우 결과도 다를 수 있다.
        - 풀 스캔 쿼리는 잘못된 결과를 반환하는 데이터까지 포함할 수 있다.
        - 공간 인덱스 레인지 스캔 쿼리는 잘못된 일부 데이터를 제외하고 더 나은 결과를 반환하는 사례가 나온다.
    - 따라서 실무에서는 다음이 중요하다.
        - 공간 함수 결과를 무조건 신뢰하지 않는다.
        - 인덱스 사용 여부에 따라 결과가 달라지는지 테스트한다.
        - 샘플 데이터뿐 아니라 실제 데이터셋으로도 검증한다.
- 12.2.4.2.2 성능 주의 사항
    - `ST_Contains()`는 `SRID=4326`보다 `SRID=3857`에서 훨씬 빠르게 동작하는 예제가 나온다.
    - 같은 연산을 100만 번 반복한 벤치마크에서 다음 경향이 확인된다.
        - `SRID=4326`은 훨씬 오래 걸린다.
        - `SRID=3857`은 몇 배 빠르다.
    - 이유는 다음과 같다.
        - 지리 좌표계는 구면 해석이 필요하다.
        - 투영 좌표계는 평면 처리로 계산 비용이 낮다.
    - 하지만 빠르다고 해서 3857이 항상 정답은 아니다.
        - 거리 계산 결과는 실제 지구상 거리와 오차가 있다.
        - 즉, 성능과 정확도 사이에서 선택해야 한다.
- 12.2.4.2.3 좌표계 변환
    - 모바일 기기 GPS 좌표는 보통 `SRID=4326`이다.
    - 웹 지도에서는 `SRID=3857`이 자주 사용된다.
    - 따라서 4326과 3857 사이의 변환이 필요할 수 있다.
    - MySQL에는 `ST_Transform()` 함수가 있지만, 예제 시점에서는 `SRID=3857` 변환을 지원하지 않는 상태로 소개된다.
    - 그래서 사용자 정의 함수로 직접 변환한다.
        - 4326을 3857로 변환하는 함수
        - 3857을 4326으로 변환하는 함수
        - 3857 두 점 간 거리 계산을 위해 내부적으로 4326으로 되돌린 뒤 `ST_Distance_Sphere()`를 호출하는 함수
    - 변환 함수에서 중요한 점은 다음과 같다.
        - 입력 SRID를 검사한다.
        - 기대한 SRID가 아니면 에러를 발생시킨다.
        - 값의 단위와 축 의미가 달라지므로 단순 숫자 복사로 처리하면 안 된다.
    - 서울과 부산 거리 예제에서는 다음이 드러난다.
        - SRID 4326 기반 거리 계산 결과가 실제 거리와 더 가깝다.
        - SRID 3857에서 단순 거리 계산한 값은 차이가 크다.
        - 따라서 지리 좌표의 실제 거리 계산은 4326 기준으로 처리해야 한다.

# **14. 스토어드 프로그램**

## **14.1 스토어드 프로그램의 장단점**

- 스토어드 프로그램은 프로시저, 함수, 트리거, 이벤트를 포함하는 절차형 SQL 구성 요소다.
- 문법과 제약은 서로 다르지만 공통적으로 데이터베이스 내부에서 절차적 처리를 수행한다.

### **14.1.1 스토어드 프로그램의 장점**

- 데이터베이스 보안 향상
    - 스토어드 프로그램 단위로 실행 권한을 부여할 수 있어 테이블 직접 접근 권한을 좁게 제어할 수 있다.
    - 특정 테이블의 읽기나 쓰기, 특정 칼럼 접근만 허용하는 형태의 세밀한 권한 제어에 유리하다.
    - 애플리케이션이 직접 SQL을 작성하지 않고 스토어드 프로그램만 호출하도록 하면 기본적인 보안 사고 가능성을 줄일 수 있다.
- 기능 추상화
    - 복잡한 계산이나 반복되는 절차를 스토어드 프로그램으로 감추면 클라이언트는 내부 SQL을 몰라도 기능을 사용할 수 있다.
    - 자바나 C/C++ 같은 언어에서 구현하던 일부 절차를 데이터베이스 안으로 옮기는 방식이다.
    - 다만 클라이언트가 직접 재사용할 라이브러리와는 성격이 다르고, DB 내부 로직으로 한정된다.
- 네트워크 소요 시간 절감
    - 여러 SQL을 순차적으로 실행해야 하는 작업을 스토어드 프로그램 한 번 호출로 줄일 수 있다.
    - 네트워크 왕복이 많은 환경에서는 이 효과가 커진다.
    - 짧고 단순한 쿼리라도 수백 번 반복되면 네트워크 비용이 누적되는데, 이를 DB 내부 절차로 묶으면 호출 횟수를 줄일 수 있다.
- 절차적 기능 구현
    - SQL만으로 표현하기 어려운 조건 분기, 반복, 단계적 처리, 중간 결과 누적 같은 절차형 로직을 구현할 수 있다.
    - 특히 여러 SQL을 순서대로 실행하면서 상태를 바꾸는 작업에 적합하다.
- 개발 역할 분리
    - DBMS 관점의 로직을 데이터베이스 내부에 두면 애플리케이션 코드와 DB 로직의 책임을 나누기 쉽다.
    - 조직에 따라 스토어드 프로그램이 별도 계층처럼 동작할 수 있다.

### **14.1.2 스토어드 프로그램의 단점**

- 낮은 처리 성능
    - 스토어드 프로그램 자체가 절차적 코드를 빠르게 실행해 주는 구조는 아니다.
    - MySQL 서버가 스토어드 프로그램 코드를 해석하고 실행해야 하므로 호출 비용이 있고, 다른 전용 언어로 작성한 프로그램보다 느릴 수 있다.
- 애플리케이션 코드와의 조화 어려움
    - 애플리케이션이 복잡해질수록 배포, 버전 관리, 테스트, 로깅, 디버깅을 DB 내부 로직과 함께 맞추기 어려워진다.
    - 서버 애플리케이션과 DB 내부 절차가 분리되면 운영 복잡도가 커질 수 있다.

## **14.2 스토어드 프로그램의 문법**

- 스토어드 프로그램은 헤더와 본문으로 나뉜다.
- 헤더에는 종류, 이름, 파라미터, 관련 옵션이 들어가고, 본문은 BEGIN ... END 또는 단일 SQL 문장으로 작성한다.
- 종류별 차이는 있지만 본문 문법은 대체로 공통이다.

### **14.2.1 예제 테스트 시 주의사항**

- 예제 실행 전에 MySQL 고유 문법과 환경 설정을 먼저 확인해야 한다.
- 프로시저나 함수 이름과 괄호 사이 공백을 허용하지 않으려면 SQL 모드 IGNORE_SPACE와 관련된 동작을 이해해야 한다.
- 32비트 운영체제 환경에서는 스택 부족 문제가 발생할 수 있다.
    - 스토어드 프로그램 예제 실행 중 Thread stack overrun 오류가 날 수 있으며, 이 경우 thread_stack 값을 키워야 한다.

### **14.2.2 스토어드 프로시저**

- 스토어드 프로시저는 여러 SQL을 하나의 독립 실행 단위로 묶는 용도에 적합하다.
- SELECT나 UPDATE 문장 안에서 식처럼 사용하는 것이 아니라 CALL로 독립 호출해야 한다.
- **14.2.2.1 스토어드 프로시저 생성 및 삭제**
    
    <img width="1030" height="192" alt="image" src="https://github.com/user-attachments/assets/252e1f79-cf08-4b38-83ea-05ae90d2ef1e" />

    - `CREATE PROCEDURE`로 생성한다.
    - 기본적으로 반환값이 없다. RETURN으로 값을 반환하는 용도가 아니다.
    - 파라미터는 세 종류가 있다.
        - IN: 입력 전용
        - OUT: 출력 전용
        - INOUT: 입력과 출력 겸용
    - ALTER PROCEDURE는 보안 속성 같은 일부 특성만 바꿀 수 있고, 파라미터나 본문을 직접 수정하지는 못한다.
    - 파라미터나 처리 로직을 바꾸려면 DROP PROCEDURE 후 다시 CREATE PROCEDURE 해야 한다.
- **14.2.2.2 스토어드 프로시저 실행**
    - CALL 프로시저명(...) 형태로 실행한다.
    - IN 파라미터는 리터럴 값 전달이 가능하다.
    - OUT이나 INOUT 파라미터는 세션 변수를 이용해 값을 주고받는 것이 일반적이다.
    - 실행 결과값은 자동 출력되지 않으므로 필요하면 세션 변수를 SELECT로 다시 조회해야 한다.
    - 자바나 C/C++ 같은 언어에서는 세션 변수를 거치지 않고 OUT/INOUT 값을 직접 받을 수 있다.
- **14.2.2.3 스토어드 프로시저의 커서 반환**
    - 명시적으로 커서를 파라미터로 전달하거나 반환하지는 못한다.
    - 대신 프로시저 내부에서 SELECT 결과를 fetch하지 않고 남겨두면 결과셋이 클라이언트로 전달된다.
    - 이 방식은 프로시저 디버깅에도 자주 쓰인다. 중간 상태를 SELECT로 찍어 보면 흐름 확인이 쉽다.
- **14.2.2.4 스토어드 프로시저 딕셔너리**
    - MySQL 8.0부터 스토어드 프로시저는 사용자에게 보이지 않는 시스템 테이블에 저장된다.
    - 조회는 information_schema.ROUTINES를 사용한다.
    - routine_schema, routine_name, routine_type 등으로 프로시저 목록을 확인할 수 있다.
    - 본문과 정의문도 ROUTINES 뷰에서 조회할 수 있다.

### **14.2.3 스토어드 함수**

- 스토어드 함수는 하나의 값을 계산해 SQL 문장 안에서 식처럼 사용할 때 적합하다.
- 프로시저와 달리 SELECT, WHERE, GROUP BY 같은 SQL 문장 일부에 포함해 사용할 수 있다.
- **14.2.3.1 스토어드 함수 생성 및 삭제**
    
    <img width="686" height="302" alt="image" src="https://github.com/user-attachments/assets/d8a21b6c-bc98-4146-8d1a-85b2f3582ff7" />

    - `CREATE FUNCTION`으로 생성한다.
    - 모든 입력 파라미터는 읽기 전용이며 IN만 존재한다.
    - 헤더에 RETURNS 타입을 반드시 명시해야 한다.
    - 본문 마지막에는 RETURN으로 헤더에 지정한 타입의 값을 돌려줘야 한다.
    - 함수 본문에서는 다음 작업을 사용할 수 없다.
        - PREPARE/EXECUTE를 이용한 프리페어드 스테이트먼트
        - 명시적 또는 묵시적 ROLLBACK/COMMIT을 유발하는 문장
        - 재귀 호출
        - 프로시저 호출
        - 결과셋을 반환하는 SQL 문장
    - ALTER FUNCTION도 본문 수정은 지원하지 않고 속성 일부만 바꾼다.
    - 입력 파라미터나 본문 변경은 DROP FUNCTION 후 재생성이 필요하다.
- **14.2.3.2 스토어드 함수 실행**
    - 일반 SQL 식처럼 SELECT sf_sum(1,2)처럼 사용한다.
    - CALL로 실행하면 MySQL은 프로시저를 찾으므로 오류가 발생한다.
    - 함수는 결과셋이 아니라 단일 반환값 중심으로 설계해야 한다.

### **14.2.4 트리거**

- 트리거는 테이블의 레코드가 저장되거나 변경될 때 자동 실행되는 스토어드 프로그램이다.
- 유효성 검사, 다른 테이블 동기화, 삭제 전 정리 작업 등에 사용한다.
- **14.2.4.1 트리거 생성**
    
    <img width="640" height="214" alt="image" src="https://github.com/user-attachments/assets/35eba0fd-fd8d-4fc0-a350-2c9ec729a278" />

    - `CREATE TRIGGER`로 생성한다.
    - BEFORE 또는 AFTER, 그리고 INSERT, UPDATE, DELETE를 조합해 이벤트 시점을 정한다.
    - MySQL 트리거는 항상 FOR EACH ROW 단위로 동작한다.
    - OLD는 변경 전 레코드, NEW는 변경 후 레코드를 가리킨다.
    - TRUNCATE나 DROP TABLE은 트리거 이벤트가 발생하지 않는다.
- **14.2.4.2 트리거 실행**
    - 명시적으로 CALL할 수 없다.
    - 테이블에 실제 DML이 발생해야 자동 실행된다.
- **14.2.4.3 트리거 딕셔너리**
    - MySQL 8.0부터 파일 기반 .TRG가 아니라 시스템 테이블에 저장된다.
    - 조회는 information_schema.TRIGGERS를 사용한다.
    - 스키마, 이름, 이벤트 조작 종류, 액션 시점, 액션 문장을 확인할 수 있다.
- **트리거 본문 제약**
    - NEW, OLD를 사용할 수 있다.
    - 외래 키로 자동 변경되는 작업에는 호출되지 않는다.
    - 복제 방식에 따라 레플리카에서 트리거 실행 방식이 달라질 수 있다.
    - 명시적 또는 묵시적 ROLLBACK/COMMIT 유발 SQL은 사용할 수 없다.
    - RETURN은 사용할 수 없고 종료 시 LEAVE를 사용한다.
    - mysql, information_schema, performance_schema의 테이블에는 생성할 수 없다.

### **14.2.5 이벤트**

- 이벤트는 특정 시각 또는 주기마다 자동 실행되는 스케줄러 기능이다.
- MySQL 서버 내부의 이벤트 스케줄러 스레드가 활성화돼 있어야 동작한다.
- **이벤트 스케줄러 활성화**
    - event_scheduler 시스템 변수를 ON으로 설정해야 한다.
    - 활성화되면 SHOW PROCESSLIST에서 event_scheduler 프로세스를 볼 수 있다.
    - MySQL은 전체 실행 이력을 오래 보관하지 않으므로, 필요하면 이벤트 내부에서 별도 로그 테이블에 기록하는 편이 좋다.
- **14.2.5.1 이벤트 생성**
    - CREATE EVENT로 생성한다.
    - 일회성 이벤트는 `ON SCHEDULE AT ...`
        
        <img width="646" height="224" alt="image" src="https://github.com/user-attachments/assets/f6517b0e-5d73-4aa5-a03e-ebc109fdbd85" />

    - 반복 이벤트는 `ON SCHEDULE EVERY ... STARTS ... ENDS ...`
        
        <img width="954" height="242" alt="image" src="https://github.com/user-attachments/assets/df9951f7-5a22-4c29-abf6-ac658e9ba4b0" />

    - 본문은 단일 SQL도 가능하고 DO BEGIN ... END 블록도 가능하다.
    - ON COMPLETION으로 완료 후 자동 삭제 여부를 정한다.
        - 기본은 완료 시 자동 삭제
        - ON COMPLETION PRESERVE면 완료 후에도 유지
    - 상태는 ENABLE, DISABLE, DISABLE ON SLAVE 개념으로 관리된다.
- **14.2.5.2 이벤트 실행 및 결과 확인**
    - 트리거처럼 강제 실행 명령은 없고, 스케줄 시각을 기다려야 한다.
    - 마지막 실행 정보와 스케줄 정보는 information_schema.EVENTS에서 확인한다.
    - 다만 EVENTS 뷰는 마지막 실행 로그만 보이므로 자세한 이력은 별도 로그 테이블이 필요하다.
- **14.2.5.3 이벤트 딕셔너리**
    - MySQL 8.0부터 시스템 테이블에 저장된다.
    - 사용자는 information_schema.EVENTS를 통해 조회한다.
    - 이벤트가 반복인지 일회성인지, 언제 생성됐는지, 언제 마지막 실행됐는지, 정의문이 무엇인지 확인할 수 있다.
    - ORIGINATOR 칼럼은 이벤트를 생성한 서버의 server_id 정보를 관리한다.

### **14.2.6 스토어드 프로그램 본문(Body) 작성**

- 본문은 공통적으로 BEGIN ... END 블록 안에서 작성한다. 여기서 트랜잭션, 변수, 제어문, 반복문 같은 절차형 요소를 사용할 수 있다.
- **14.2.6.1 BEGIN … END 블록과 트랜잭션**
    - 하나의 BEGIN ... END 블록은 또 다른 BEGIN ... END를 중첩 포함할 수 있다.
    - 블록 안의 BEGIN은 트랜잭션 시작이 아니라 단순 블록 시작으로 해석된다.
    - 트랜잭션 시작은 START TRANSACTION을 써야 한다.
    - 프로시저와 이벤트에서는 트랜잭션을 사용할 수 있다.
    - 함수와 트리거에서는 트랜잭션 제어를 사용할 수 없다.
    - 프로시저 내부에서 COMMIT이나 ROLLBACK을 수행하면 호출자 입장에서 트랜잭션을 조절할 수 없다.
    - 프로시저를 오토커밋 모드에서 호출하면 내부 SQL은 각각 커밋될 수 있으므로 주의가 필요하다.
- **14.2.6.2 변수**
    
    <img width="808" height="486" alt="image" src="https://github.com/user-attachments/assets/3466856d-53ee-4a08-84bb-5d26a3ff4199" />

    - 사용자 변수와 로컬 변수는 다르다.
    - 로컬 변수는 `DECLARE` 변수명 타입 [DEFAULT 값]으로 선언한다.
    - 값 할당은 SET 또는 SELECT ... INTO ...를 사용한다.
    - 중첩 블록에서는 가장 가까운 바깥 블록의 로컬 변수를 참조한다.
- **14.2.6.3 제어문**
    - **14.2.6.3.1 IF … ELSEIF … ELSE … END IF**
        - 일반 프로그래밍 언어의 if 문과 유사하다.
        - END IF로 종료하며 끝에 구분자 표기가 필요하다.
        - 함수나 프로시저 안에서 조건 분기에 사용한다.
    - **14.2.6.3.2 CASE WHEN … THEN … ELSE … END CASE**
        - switch와 비슷하지만 두 형태가 있다.
            - CASE 값 WHEN 비교값 THEN ...
            - CASE WHEN 조건식 THEN ...
        - 각 WHEN ... THEN 뒤에 별도 BREAK는 필요 없다.
        - END CASE로 종료한다.
    - **14.2.6.3.3 반복 루프**
        - LOOP, REPEAT, WHILE을 사용할 수 있다.
        - LOOP는 탈출 조건을 내부에서 명시해야 하며, LEAVE로 빠져나온다.
        - REPEAT는 본문을 먼저 실행하고 UNTIL 조건이 참이면 종료한다.
        - WHILE은 조건을 먼저 검사하고 참일 때만 반복한다.
